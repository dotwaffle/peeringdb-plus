<objective>
Replace the per-row FK backfill HTTP pattern (`?since=1&id__in=N` per missing parent ID) with a dataloader-style batched pattern (`?since=1&id__in=1,2,3,…` per parent type per chunk). Steady state in production is ~10-16 distinct misses/cycle (acceptable today), but truncate-and-resync recovery, long downtime catch-up, or unusual upstream churn could push to hundreds-to-thousands of distinct missing parents — exactly the scenario that bricked v1.18.2 by hitting upstream's `API_THROTTLE_REPEATED_REQUEST` cap (1MB / 10/min). Batching collapses N HTTP requests into ⌈N/100⌉, eliminating the throttle exposure.

Purpose: protect the catch-up / recovery code path against upstream throttle by preserving N-to-1 batching wherever the chunk pre-pass can see N missing IDs of the same parent type at once. Recursion stays bounded by the existing per-cycle dedup cache (each (type, id) fires exactly once).

Output: new `peeringdb.Client.FetchByIDs`, refactored `fkBackfillBatch` entry point, a `prefetchMissingParentsForChunk` hook in `dispatchScratchChunk`, and 5 new tests. All existing tests continue to pass against the preserved `fkBackfillParent` thin wrapper.
</objective>
