# Project Research Summary

**Project:** PeeringDB Plus -- ConnectRPC / gRPC API Surface Milestone
**Domain:** Adding a gRPC/ConnectRPC API surface to an existing read-only PeeringDB data mirror
**Researched:** 2026-03-24
**Confidence:** HIGH

## Executive Summary

PeeringDB Plus already serves data through GraphQL, REST (OpenAPI), PeeringDB-compatible REST, and a web UI. This milestone adds a fourth API surface: gRPC via ConnectRPC. The recommended approach uses ConnectRPC rather than standard gRPC, because ConnectRPC handlers are plain `http.Handler` values that mount directly on the existing `http.NewServeMux()` alongside all other routes. This eliminates the need for a separate gRPC server, a second port, or any changes to `fly.toml`. The proto generation pipeline uses entproto to generate `.proto` files from the existing ent schemas, then `buf generate` with `protoc-gen-connect-go` to produce Go types and handler interfaces. Service implementations are hand-written (not auto-generated by `protoc-gen-entgrpc`) to maintain control over query behavior and error mapping for this read-only mirror.

The most consequential architectural constraint is that the LiteFS proxy between Fly.io's edge and the application is HTTP/1.1 only. The native gRPC wire protocol (which requires HTTP/2) cannot traverse this proxy. ConnectRPC solves this because its Connect protocol and gRPC-Web protocol both work over HTTP/1.1 for unary RPCs. Since all operations are read-only Get/List, no streaming is needed, and HTTP/1.1 is fully sufficient. This is not a compromise -- Connect protocol is curl-able with JSON, browser-compatible, and supported by all major gRPC client libraries.

The primary implementation risks center on the entproto annotation process: 13 ent schemas with 30-40+ fields each need `entproto.Field(N)` annotations with permanently-assigned field numbers. JSON fields (`social_media`, `info_types`) are not supported by entproto and require manual proto definitions. Field number collisions between schema fields and edges must be prevented through systematic numbering conventions (fields 2-N, edges 100+). The `buf breaking` CI check is essential to prevent accidental field number changes from silently corrupting client data. Secondary risks include CORS configuration gaps (gRPC-Web/Connect headers not in the current allowlist) and OTel double-instrumentation (otelhttp + otelconnect on the same request path).

## Key Findings

### Recommended Stack

ConnectRPC v1.19.1 is the centerpiece. It serves Connect, gRPC, and gRPC-Web protocols simultaneously from standard `http.Handler` values, requiring no separate server or port. All dependencies are version-compatible with the project's Go 1.26.1 and OTel v1.42.0. See [STACK.md](STACK.md) for full version matrix and alternatives considered.

**Core technologies:**
- **ConnectRPC v1.19.1**: RPC framework -- handlers are `http.Handler`, mount on existing mux, support triple-protocol
- **entproto (via entgo.io/contrib v0.7.1)**: Proto file generation from ent schemas -- already in `go.mod`, generates `.proto` from annotated schemas
- **buf CLI v1.66.0+**: Protobuf toolchain -- linting, breaking change detection, code generation; replaces raw `protoc`
- **protoc-gen-connect-go v1.19.1**: Code generator -- produces ConnectRPC handler interfaces and clients from `.proto` files
- **otelconnect v0.9.0**: OTel interceptor -- RPC-level tracing and metrics per OTel semantic conventions
- **grpcreflect v1.3.0 + grpchealth v1.4.0**: gRPC ecosystem services -- reflection for tooling, health checks for monitoring

**Not needed (explicit exclusions):**
- `google.golang.org/grpc` as direct dependency (ConnectRPC replaces it)
- `protoc-gen-go-grpc` (ConnectRPC uses `protoc-gen-connect-go` instead)
- `protoc-gen-entgrpc` (generates standard gRPC impls, not ConnectRPC handlers)
- `golang.org/x/net/http2/h2c` (superseded by Go 1.24+ stdlib, and h2c is not needed through LiteFS proxy)

### Expected Features

See [FEATURES.md](FEATURES.md) for complete feature landscape, decision matrix, and dependency graph.

**Must have (table stakes):**
- Get by ID for all 13 entity types -- fundamental read operation
- List with keyset pagination for all 13 types -- `page_size` + `page_token` per AIP-158
- Proto message definitions for all 13 types with correct field mappings
- Standard gRPC error codes (`NOT_FOUND`, `INVALID_ARGUMENT`, `UNAVAILABLE`)
- Server reflection -- enables grpcurl, grpcui, Postman discovery
- gRPC health check service -- required for monitoring and load balancer integration
- OTel instrumentation on all RPCs -- project constraint
- Read-only enforcement -- only `MethodGet | MethodList`, no mutation RPCs
- buf toolchain integration -- linting, breaking change detection in CI

**Should have (differentiators):**
- ConnectRPC triple-protocol support -- curl-able JSON, browser-compatible, gRPC-Web native
- SyncStatus custom RPC -- data freshness visibility for automation clients
- Typed filter fields on List RPCs -- filter by ASN, country, name, org_id, status
- Edge/relationship traversal in proto messages -- nested entities to avoid N+1 calls

**Defer (v2+):**
- Server streaming for bulk export -- pagination covers the use case
- Field masks (AIP-157) -- marginal benefit for local SQLite reads
- AIP-160 filter expression parser -- too complex; typed filter fields are sufficient
- Client SDK generation/publishing -- publish `.proto` files, let consumers generate

### Architecture Approach

ConnectRPC handlers mount on the existing `http.NewServeMux()` at paths like `/peeringdb.v1.NetworkService/`. They share the middleware stack (Recovery, otelhttp, Logging, CORS, Readiness) with REST and GraphQL. The proto generation pipeline is two-stage: `go generate ./ent/...` produces `.proto` files via entproto, then `buf generate` produces Go types and ConnectRPC interfaces. Service implementations live in `internal/connectrpc/` and query the ent client directly. No `fly.toml` changes required -- the Connect protocol works over HTTP/1.1 through LiteFS proxy unchanged. See [ARCHITECTURE.md](ARCHITECTURE.md) for full component diagram, data flow, and patterns.

**Major components:**
1. **ent schema annotations** (`ent/schema/*.go`) -- `entproto.Message()`, `entproto.Field(N)`, `entproto.Service()` on all 13 schemas
2. **Proto generation pipeline** (`buf.yaml`, `buf.gen.yaml`, `ent/entc.go`) -- entproto + buf generate chain producing `gen/peeringdb/v1/`
3. **Service implementations** (`internal/connectrpc/`) -- hand-written Get/List handlers per entity type, ent-to-proto conversion functions
4. **Handler registration** (`cmd/peeringdb-plus/main.go`) -- mount handlers on mux with otelconnect interceptor, grpcreflect, grpchealth

### Critical Pitfalls

See [PITFALLS.md](PITFALLS.md) for all 13 pitfalls with detailed prevention strategies and code examples.

1. **LiteFS proxy is HTTP/1.1 only** -- Do not set `h2_backend = true` in fly.toml. Use ConnectRPC's Connect protocol (HTTP/1.1) exclusively. Native gRPC wire protocol is not supported. This is acceptable because all RPCs are unary and Connect protocol is functionally equivalent.

2. **Proto field numbers are permanent** -- Once assigned via `entproto.Field(N)`, numbers must never change. Use `buf breaking` in CI from day one. Use numbering gaps (fields 2-N, edges 100+) to allow future additions. Reserve removed field numbers.

3. **OTel double-instrumentation** -- otelhttp (mux-level) and otelconnect (handler-level) both create spans for gRPC requests, producing duplicate metrics. Filter ConnectRPC paths from otelhttp using `otelhttp.WithFilter()`, or mount ConnectRPC on a sub-mux without otelhttp.

4. **CORS missing gRPC-Web/Connect headers** -- Current CORS config only allows `Content-Type` and `Authorization`. Must add `X-Grpc-Web`, `Grpc-Timeout`, `Connect-Protocol-Version`, `Connect-Timeout-Ms`, and expose `Grpc-Status`, `Grpc-Message`, `Grpc-Status-Details-Bin`. Browser clients fail silently without this.

5. **entproto JSON field types not supported** -- `social_media` (6 schemas) and `info_types` (Network) use `field.JSON` which entproto cannot map. Define `SocialMedia` as a separate proto message manually. Add `repeated string info_types` manually. Proto files will be partially generated, partially hand-maintained.

## Implications for Roadmap

Based on research, suggested phase structure:

### Phase 1: Schema Annotations and Proto Generation Pipeline

**Rationale:** Everything depends on proto definitions existing. This phase establishes the foundation that all subsequent phases build on. The entproto annotation process is the highest-risk activity (JSON field blockers, field number assignment permanence, edge number collisions) and must be validated early.
**Delivers:** Annotated ent schemas, generated `.proto` files, `buf.yaml`/`buf.gen.yaml` config, generated Go types and ConnectRPC interfaces. The `buf lint` and `buf breaking` checks in CI.
**Addresses:** Proto message definitions (table stakes), buf toolchain integration (table stakes)
**Avoids:** Field number permanence pitfall (#2), edge number collisions (#9), JSON field type failures (#10), missing annotations (#13), pipeline ordering issues (#11), enum naming conflicts (#5)

### Phase 2: First Service + Handler Mounting + Infrastructure

**Rationale:** Prove the pattern works end-to-end with one entity before committing to all 13. This phase validates ConnectRPC handler mounting on the existing mux, otelconnect instrumentation, CORS configuration, readiness middleware updates, and the LiteFS proxy HTTP/1.1 constraint. Pick Organization (simplest entity, fewest edges) as the proof-of-concept.
**Delivers:** One working ConnectRPC service (Organization Get + List), otelconnect interceptor, grpcreflect, grpchealth, updated CORS middleware, updated readiness bypass list. End-to-end verification with `buf curl` or grpcurl.
**Addresses:** Get/List operations (table stakes), server reflection (table stakes), health checks (table stakes), OTel instrumentation (table stakes), error handling (table stakes), read-only enforcement (table stakes), ConnectRPC triple-protocol (differentiator)
**Avoids:** LiteFS h2c pitfall (#1), CORS gaps (#4), OTel double-instrumentation (#3), readiness blocking health checks (#7), route conflicts (#12), Fly.io ALPN misconfiguration (#8)

### Phase 3: Remaining Services + SyncStatus

**Rationale:** With the pattern proven in Phase 2, implement the remaining 12 entity services. Each follows the same pattern (Get + List + conversion functions) and can be done in parallel. Add the custom SyncStatus RPC which provides data freshness visibility. This is mechanical work with low risk.
**Delivers:** All 13 entity services with Get/List operations, SyncStatus custom RPC, comprehensive tests.
**Addresses:** Full entity coverage (table stakes), SyncStatus (differentiator)
**Avoids:** Wrapper type ergonomics (#6) -- accept wrapper types, document in API docs

### Phase 4: Filtering and Edge Traversal (Optional)

**Rationale:** Once the basic API is complete and stable, add differentiating features based on user demand. Typed filter fields on List RPCs (filter by ASN, country, etc.) are the highest-value addition. Edge/relationship traversal in proto messages reduces N+1 calls for power users. Both require changes to request/response message definitions, so `buf breaking` CI check from Phase 1 protects the contract.
**Delivers:** Filtered List RPCs with typed filter fields, nested entity messages for key relationships.
**Addresses:** Filtering on List RPCs (differentiator), edge/relationship traversal (differentiator)
**Avoids:** Field number changes on existing messages (#2) -- use `buf breaking` to validate

### Phase Ordering Rationale

- Phase 1 must come first because proto definitions are a build-time dependency for all service code.
- Phase 2 proves the full integration path (schema -> proto -> handler -> mux -> middleware -> client) with minimum scope. Every infrastructure concern (CORS, readiness, OTel, h2c constraint) is resolved here before scaling to 13 services.
- Phase 3 is deliberately separated from Phase 2 to isolate integration risk from volume risk. Once the pattern works for one entity, the remaining 12 are mechanical.
- Phase 4 is optional and demand-driven. The basic Get/List API is complete after Phase 3.

### Research Flags

Phases likely needing deeper research during planning:
- **Phase 1:** entproto annotation behavior with the project's specific schema types (FlexDate, FlexInt, custom enums). JSON field workaround must be validated with actual code generation. The `entproto.SkipGenFile()` + `buf generate` pipeline is not documented as a combined pattern -- needs hands-on validation.
- **Phase 4:** Filtering implementation requires translating typed filter fields to ent predicates. No entproto support exists for query predicates. This is custom code with no established pattern.

Phases with standard patterns (skip research-phase):
- **Phase 2:** ConnectRPC handler mounting, otelconnect, grpcreflect, grpchealth are all well-documented with canonical examples. CORS updates are straightforward.
- **Phase 3:** Repetitive implementation of the pattern established in Phase 2. No new technical decisions.

### Gaps to Address

- **entproto + custom types (FlexDate, FlexInt):** The ent schemas use custom types in `ent/schema/types.go`. entproto's behavior with these types is undocumented. Must validate during Phase 1 by running `go generate` on the first annotated schema.
- **entproto edge handling for circular references:** Organization -> Network -> Organization is a circular edge. entproto should handle this within a single proto package, but needs validation.
- **Wrapper types vs proto3 `optional`:** entproto generates wrapper types for Optional/Nillable fields. Proto3 `optional` keyword (available since protobuf 3.15) would be more ergonomic. Unclear if entproto supports the `optional` keyword -- check during Phase 1.
- **`buf breaking` with entproto-generated protos:** The `buf breaking` check compares against the git base branch. If entproto regenerates proto files with different formatting or ordering (but same content), `buf breaking` may flag false positives. Needs validation.
- **Performance under LiteFS FUSE overhead:** ConnectRPC adds protobuf serialization on top of ent queries on top of SQLite reads through LiteFS FUSE mount. The combined overhead for List operations (50K+ records) should be benchmarked during Phase 3.
