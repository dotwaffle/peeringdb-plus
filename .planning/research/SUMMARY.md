# Project Research Summary

**Project:** PeeringDB Plus v1.1 (REST API & Observability)
**Domain:** Read-only PeeringDB data mirror -- adding REST API surfaces and completing observability
**Researched:** 2026-03-22
**Confidence:** HIGH

## Executive Summary

PeeringDB Plus v1.1 adds two REST API surfaces and completes the observability story left incomplete in v1.0. The project already has a working data sync pipeline, GraphQL API, and basic OTel tracing. This milestone fixes known tech debt (sync metrics registered but never recorded, no HTTP client tracing) and adds both a modern entrest-generated REST API and a PeeringDB-compatible REST layer for drop-in replacement of the upstream PeeringDB API. The recommended approach requires zero new runtime dependencies -- entrest is a build-time code generator, and all OTel packages are already in `go.mod`.

The architecture calls for two independent REST API surfaces sharing the same ent data layer: a modern OpenAPI REST API at `/rest/` generated by entrest (standard pagination, filtering, OpenAPI spec), and a hand-written PeeringDB compatibility layer at `/api/` that reproduces PeeringDB's exact response envelope (`{"meta": {}, "data": [...]}`), URL paths (`/api/net`, `/api/fac`), and Django-style query parameter filters (`__contains`, `__in`, `__lt`). This dual-surface approach is strongly recommended over trying to force one API surface to serve both purposes. The entrest-generated API and PeeringDB compat layer have fundamentally different response formats, pagination models, and query parameter conventions that cannot be reconciled in a single handler.

The primary risks are: (1) PeeringDB's Django-style filter syntax is much deeper than URL path compatibility -- consumers rely heavily on `__contains`, `__in`, and numeric comparison filters that must be translated to ent predicates; (2) the MeterProvider is not initialized in the OTel pipeline, meaning all metric recordings silently go to a no-op unless this is fixed first; (3) entrest warns of potential breaking changes despite being v1.x, but the REST surface is independently useful and can be replaced with hand-rolled handlers if needed. All three risks have clear mitigations.

## Key Findings

### Recommended Stack

No new runtime dependencies are needed for v1.1. The only new direct dependency is `github.com/lrstanley/entrest` v1.0.2, which is a build-time code generator (its transitive dependencies do not appear in the compiled binary). The otelhttp package for HTTP client tracing and the OTel metric API are both already in `go.mod` from v1.0 work.

**Core technologies (v1.1 additions):**
- **entrest v1.0.2**: REST API code generation from ent schemas -- generates read-only HTTP handlers and OpenAPI 3.1 spec using stdlib `net/http` ServeMux (MEDIUM confidence due to "expect breaking changes" warning)
- **otelhttp v0.67.0 (existing)**: Wraps HTTP client transport for automatic outbound request tracing -- one-line change to `NewClient()` (HIGH confidence)
- **OTel metric API v1.42.0 (existing)**: Counters, histograms, and gauges for sync metrics -- `Float64Gauge` stabilized in v1.31.0 (HIGH confidence)

### Expected Features

**Must have (table stakes):**
- OTel HTTP client tracing on PeeringDB API calls with span hierarchy (full-sync > sync-type > HTTP request)
- MeterProvider initialization (currently missing -- prerequisite for all metrics)
- Sync metrics actually recorded (duration, object counts, errors, last success timestamp)
- entrest-generated read-only REST endpoints for all 13 object types with OpenAPI spec
- PeeringDB-compatible REST layer with exact URL paths, response envelope, and query parameter support

**Should have (differentiators):**
- Correlated traces + metrics across sync lifecycle via exemplars
- otelhttp server-side request tracing for all incoming API requests
- Dual REST surfaces: modern at `/rest/`, PeeringDB-compatible at `/api/`
- Generated OpenAPI spec served at `/rest/openapi.json` (PeeringDB's own spec is broken)

**Defer (v1.2+):**
- PeeringDB `depth` support beyond 0 and 2 (depths 1, 3, 4 are rarely used)
- `_set` field expansion matching PeeringDB's nested format
- `since` parameter in compat layer
- Rate limiting (defeats the purpose of a fast mirror)
- Write operations (read-only mirror)
- PeeringDB authentication emulation

### Architecture Approach

The v1.1 additions integrate into the existing single-binary architecture with minimal disruption. All three API surfaces (GraphQL, entrest REST, PeeringDB compat) share the same `ent.Client` and SQLite database. The existing middleware stack (Recovery > otelhttp > Logging > CORS > readinessMiddleware) applies to all new routes without modification. The readiness gate automatically prevents serving data before first sync completes.

**Major components:**
1. **OTel HTTP client transport** (`internal/peeringdb/client.go`) -- wraps outbound HTTP client with `otelhttp.NewTransport` for automatic span creation on PeeringDB API calls
2. **Expanded sync metrics** (`internal/otel/metrics.go` + `internal/sync/worker.go`) -- initialize MeterProvider, register 7 metric instruments, wire recording calls into sync loop at per-step granularity
3. **entrest REST API** (`ent/entc.go` + `ent/rest/` generated + 13 schema annotation updates) -- code-generated read-only REST handlers with pagination, filtering, sorting, and OpenAPI spec
4. **PeeringDB compat layer** (`internal/pdbcompat/` new package) -- hand-written handlers mapping PeeringDB URL paths, response envelope, and Django-style query filters to ent queries

### Critical Pitfalls

1. **MeterProvider not initialized** -- All metric recordings silently go to OTel no-op. Must initialize MeterProvider alongside TracerProvider in `internal/otel/provider.go` before writing any metric recording code. This is the most dangerous pitfall because everything compiles and tests pass while metrics are silently dropped.

2. **Span hierarchy wrong with retries** -- otelhttp creates one span per HTTP round-trip, so retries appear as flat sibling spans instead of children of a logical operation. Must create explicit parent spans in `FetchAll()` per object type, child spans per page in `doWithRetry()`, with transport spans nested below. Add `http.request.resend_count` attribute per OTel conventions.

3. **PeeringDB query parameter depth** -- PeeringDB consumers rely on Django-style filter suffixes (`__contains`, `__in`, `__lt`, `__gte`) that are not standard REST. The compat layer must parse these suffixes and translate them to ent Where predicates. Start with exact match and `__in`, add others incrementally.

4. **entrest write handlers on read-only database** -- entrest generates CRUD by default. Must whitelist read-only operations via `entrest.WithIncludeOperations(OperationRead, OperationList)` on every schema. Add defense-in-depth middleware rejecting non-GET requests to REST paths. Verify generated OpenAPI spec contains no write operations in CI.

5. **Two API surfaces, not one** -- Do not attempt to make entrest produce PeeringDB-format responses. They have fundamentally different response envelopes, pagination models, and query parameters. Build as separate handler trees querying the same ent client.

## Implications for Roadmap

Based on research, suggested phase structure:

### Phase 1: Observability Foundations

**Rationale:** Fixes known v1.0 tech debt first. Provides operational visibility before adding new API surfaces. Zero new dependencies. Smallest scope, highest confidence.
**Delivers:** Full OTel observability -- HTTP client traces for PeeringDB API calls, MeterProvider initialization, and sync metrics actually being recorded.
**Addresses:** OTel HTTP client tracing (table stakes), MeterProvider initialization (table stakes), sync metrics wiring (table stakes)
**Avoids:** Pitfall #4 (no-op MeterProvider) by initializing MeterProvider first; Pitfall #1 (span hierarchy) by designing span tree before instrumenting; Pitfall #7 (duplicate instrument registration) by creating instruments at Worker construction time; Pitfall #9 (transport layer violations) by keeping retry logic outside RoundTripper

**Scope:**
- Initialize MeterProvider in `internal/otel/provider.go` alongside TracerProvider
- Wrap PeeringDB HTTP client transport with `otelhttp.NewTransport` in `NewClient()`
- Add PeeringDB-specific span attributes (object type, page number) in `FetchAll()`
- Wire existing `SyncDuration` and `SyncOperations` metrics to `Record()` / `Add()` calls
- Add new per-type metrics: objects synced, step duration, deletes, last success timestamp, errors
- Record metrics per sync step (not just at completion) per Pitfall #13

### Phase 2: entrest REST API

**Rationale:** Prerequisite for Phase 3 (project must compile with new schema annotations). entrest does most work via code generation -- same pattern as entgql in v1.0. Medium complexity but well-understood. The modern REST API is independently valuable even without the PeeringDB compat layer.
**Delivers:** Read-only REST API at `/rest/` with auto-generated OpenAPI spec, pagination, filtering, sorting, and edge eager-loading for all 13 object types.
**Uses:** entrest v1.0.2 (new dependency), existing ent schemas, stdlib ServeMux
**Implements:** Dual API Surface pattern (Pattern 3 from ARCHITECTURE.md)
**Avoids:** Pitfall #5 (extension conflicts) by testing codegen with both extensions before building handlers; Pitfall #8 (write handlers on read-only DB) by whitelisting read operations; Pitfall #10 (internal field exposure) by per-field annotation review

**Scope:**
- Add entrest dependency to `go.mod`
- Add entrest extension to `ent/entc.go` alongside existing entgql extension
- Annotate all 13 ent schemas with entrest operations, field filters, sortable fields, and JSON schema definitions for complex fields
- Run code generation and verify both GraphQL and REST compile together
- Mount REST handler at `/rest/` with CORS middleware
- Serve OpenAPI spec at `/rest/openapi.json`
- Add otelhttp server-side tracing wrapper for all incoming requests

### Phase 3: PeeringDB Compatibility Layer

**Rationale:** Highest complexity, highest risk of undocumented PeeringDB API quirks. Benefits from Phase 1 observability for debugging. Queries ent directly (does not depend on entrest at runtime). The Django-style filter-to-predicate translation is the hardest single piece of the milestone.
**Delivers:** Drop-in PeeringDB API replacement at `/api/` with exact URL paths, response envelope, and query parameter compatibility for existing PeeringDB consumers.
**Uses:** ent client directly, stdlib `net/http`, existing PeeringDB types from `internal/peeringdb/types.go`
**Implements:** Compatibility Layer as Adapter pattern (Pattern 4 from ARCHITECTURE.md)
**Avoids:** Pitfall #2 (response format mismatch) by building separate handler tree; Pitfall #3 (field name gaps) by reusing existing PeeringDB types for serialization; Pitfall #6 (query parameter depth) by prioritizing common filters first; Pitfall #11 (depth complexity) by starting with depth=0 only

**Scope:**
- Create `internal/pdbcompat/` package with handler, response, filter, depth, and types modules
- Implement URL routing for all 13 PeeringDB object types (`/api/{type}` and `/api/{type}/{id}`)
- Implement response envelope serialization (`{"meta": {}, "data": [...]}`)
- Implement query parameter parsing: `limit`, `skip`, `fields` first
- Implement Django-style field filters: exact match and `__in` first, then `__contains`, `__startswith`, numeric comparisons incrementally
- Implement `depth=0` (flat, default) and `depth=2` (eager-loaded edges) -- defer depth 1/3/4
- Golden file tests comparing compat layer output against real PeeringDB API responses

### Phase Ordering Rationale

- **Phase 1 before Phase 2**: Observability fixes provide diagnostic capability for all subsequent work. The MeterProvider initialization is a hard prerequisite for all metrics. No schema changes or code generation needed.
- **Phase 2 before Phase 3**: Schema annotations must be added and code must compile with both extensions before the compat layer can be built. Understanding what entrest generates informs compat layer design decisions. The entrest REST API is independently valuable.
- **Phase 3 last**: Most complex feature with highest risk of PeeringDB API quirks requiring iteration. Benefits from observability (Phase 1) being in place to diagnose issues. The filter-to-predicate translation is a large surface area (13 types x multiple filter types).

### Research Flags

Phases likely needing deeper research during planning:
- **Phase 3 (PeeringDB Compat Layer):** The Django-style filter syntax has undocumented edge cases. Must test against real PeeringDB API responses from all 13 object types. The `depth` parameter behavior is documented but the exact response shapes for each type at each depth level require empirical validation against the PeeringDB Python source or live API.

Phases with standard patterns (skip research-phase):
- **Phase 1 (Observability):** All patterns are well-documented by OpenTelemetry. otelhttp transport wrapping is canonical (2,691 importers). MeterProvider setup follows the same pattern as TracerProvider (already implemented). HIGH confidence.
- **Phase 2 (entrest REST API):** entrest has a getting-started guide and annotation reference. The pattern mirrors entgql integration from v1.0. The only uncertainty is extension co-existence with entgql, which is resolved by running `go generate` early. MEDIUM-HIGH confidence.

### Gaps to Address

- **entrest + entgql co-existence**: Must be validated by running `go generate` with both extensions before writing any handler code. No documentation confirms they work together. Risk is MEDIUM but mitigation (test codegen first) is straightforward.
- **PeeringDB depth parameter exact behavior per type**: Documentation describes general depth behavior, but each object type has different edges and the exact response shape at each depth level may have undocumented quirks. Validate with golden file tests against live PeeringDB API.
- **entrest pagination format**: The exact query parameter names and response pagination metadata format are not fully documented. Must inspect generated code to understand the format and document it for API consumers.
- **PeeringDB `since` parameter semantics**: Deferred to v1.2+ but worth understanding during Phase 3 planning -- this is a critical parameter for incremental sync consumers.
- **LiteFS behavior with multiple concurrent read queries**: The entrest and pdbcompat surfaces add concurrent read load. LiteFS handles this well (reads are local), but verify under load during Phase 2 testing.
