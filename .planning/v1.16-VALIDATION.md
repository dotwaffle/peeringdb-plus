## Core design discovery (needed before interpreting claims)

PeeringDB does **not** use DRF `filterset_class` / `django-filter`. Filtering is implemented in one place — `ModelViewSet.get_queryset` at `src/peeringdb_server/rest.py:443-755` — via a custom `query_params` loop. There are two filter mechanisms:

- **Path A — per-serializer whitelist.** `Serializer.prepare_query(qset, **kwargs)` (called at `rest.py:469`). Each serializer whitelists a small set of pseudo-fields (e.g. `ix_id`, `country`, `name`) that drive hand-written `Model.related_to_<x>` helpers in `models.py`.
- **Path B — generic auto-FK traversal.** `ModelSerializer.queryable_relations()` at `serializers.py:754-780` reflects over `_meta.get_fields()` and exposes every `<fk_name>__<related_fld>` (minus `FILTER_EXCLUDE` at `serializers.py:128-157`) as a filterable field. These join `field_names` at `rest.py:504-507` and are applied in the main loop (`rest.py:587-662`).

Replication in v1.16: Phase 70 implements both paths (D-01 Path A via ent annotation + codegen, D-02 Path B via ent introspection).

## Invalid pdbfe claims (fed into `docs/API.md § Validation Notes` by Phase 72)

These 5 claims by pdbfe are demonstrably wrong against upstream source. Phase 72 D-05 documents them with `peeringdb/peeringdb@99e92c72` file:line refs so future auditors don't re-derive.

| # | pdbfe claim | Reality | Upstream evidence |
|---|-------------|---------|-------------------|
| 1 | `net?country=NL` works as a direct filter | **FALSE** — Network has no direct `country` field. Must use `net?org__country=NL` (Path B traversal). Pattern #10 in pdbfe's peeringdb-api-patterns.md is itself wrong. | `serializers.py:2938-2992` (NetworkSerializer.prepare_query doesn't whitelist country); Network model has no country field |
| 2 | `limit=0` returns count-only with empty data | **FALSE** — returns all rows, no envelope change. | `rest.py:494-497` (`qset[skip:]` no upper bound) |
| 3 | Default ordering is `ORDER BY id ASC` | **FALSE** — default is `(-updated, -created)`. | `django-handleref/models.py:95-101` (`class Meta: ordering = ("-updated", "-created")`) |
| 4 | Unicode folding comes from MySQL `utf8_general_ci` collation | **FALSE** — folding is Python `unidecode.unidecode(v)` on the Python side BEFORE hitting SQL. Dev stack uses `utf8_unicode_ci` (not `utf8_general_ci`). Production collation unknown from repo. | `rest.py:576` (`v = unidecode.unidecode(v)`); `Ctl/dev/docker-compose.yml` (MySQL 8 `--collation-server=utf8_unicode_ci`) |
| 5 | Filter surface is DRF `filterset_class` | **FALSE** — no django-filter, no filterset. Custom `query_params` loop at `rest.py:443-755` with two paths (Path A whitelist + Path B auto-expose). | `rest.py:443-755` (custom get_queryset); `serializers.py:754-780` (queryable_relations) |
