    Important: the `updated_since` predicate goes into the `predicates` slice so it composes with existing filters AND affects the count query. The `since_id` sets `lastID` directly (it's a cursor, not a filter predicate) so it does NOT affect the count query -- this is intentional: `grpc-total-count` should reflect the total matching records regardless of resume position.

    Wait -- reconsider: should `since_id` affect `grpc-total-count`? The use case is resume: client received records up to ID 500, reconnects with since_id=500. The total count should reflect how many records REMAIN (not how many existed originally). So `since_id` should ALSO be added as a predicate for the count query. Actually no -- the count query already applies all filter predicates, and `since_id` is not a filter, it's a cursor. The total count header tells the client "there are N total matching records". For resume, the client already knows how many it received. Adding IDGT to the count would make the header value change between calls, which is confusing. Keep `since_id` as cursor only (sets lastID), NOT as a count predicate.

    Actually, re-reading the CONTEXT.md: "since_id translates to WHERE id > since_id predicate -- composes naturally with keyset pagination". This says it IS a predicate. And for consistency, if a consumer sends since_id, they want to know how many records match their complete filter set. Let me add it as a predicate on the count query too. This way grpc-total-count = number of records the client will actually receive.

    Final decision: Add `since_id` as BOTH:
    1. A predicate in the count query: `predicates = append(predicates, network.IDGT(int(*req.SinceId)))` -- so count reflects records that will be streamed
    2. The initial lastID for keyset pagination: `lastID = int(*req.SinceId)` -- so first batch starts after since_id

    This is consistent because the keyset loop uses `WHERE id > lastID` which matches `IDGT(since_id)`.

    Updated pattern:
    ```go
    // After existing entity-specific filter predicates:

    // Resume and incremental filter support.
    if req.SinceId != nil {
        predicates = append(predicates, network.IDGT(int(*req.SinceId)))
    }
    if req.UpdatedSince != nil {
        predicates = append(predicates, network.UpdatedGT(req.UpdatedSince.AsTime()))
    }

    // ... count query uses predicates (includes since_id and updated_since) ...

    // Keyset loop -- start from since_id if provided.
    lastID := 0
    if req.SinceId != nil {
        lastID = int(*req.SinceId)
    }
    ```

    The IDGT predicate in the count is technically redundant with lastID initialization, but keeps the count accurate. The keyset loop already applies `WHERE id > lastID` per batch, and `predicates` are also applied, so both work together correctly without double-filtering (the loop's IDGT(lastID) will always be >= IDGT(sinceId) since lastID starts at sinceId and only increases).
