### Locked Decisions
- **D-01:** REST API mounted at `/rest/v1/` prefix -- namespace separation from GraphQL (/graphql) and PeeringDB compat (/api/), plus versioning
- **D-02:** Use entrest default resource naming from ent schema names (e.g. Network -> /networks, InternetExchange -> /internet-exchanges) -- no custom overrides

- **D-08:** JSON fields (social_media, info_types) included as-is in REST responses -- raw JSON arrays/objects

## Summary

This phase adds entrest v1.0.2 as a second ent extension (alongside the existing entgql) to generate read-only REST API handlers and an OpenAPI 3.1 specification for all 13 PeeringDB object types. The entrest library generates HTTP handlers using Go 1.22+ stdlib ServeMux routing, which matches the project's existing approach. The generated server provides a `Handler()` method returning a standard `http.Handler` that mounts on the project's shared mux.

The integration follows the identical pattern used for entgql in v1.0: add the extension to `ent/entc.go`, add annotations to each schema's `Annotations()` method, run `go generate`, and mount the resulting handler. The key difference is that entrest generates REST handlers rather than GraphQL resolvers, and produces an OpenAPI specification alongside the handler code. Both extensions operate on the same ent schema graph but generate independent code files (entgql generates `gql_*.go`, entrest generates files under `ent/rest/`).

The primary risk is entrest + entgql coexistence -- no documentation confirms they work together, but they use separate annotation namespaces and generate separate output files. This must be validated by running `go generate` early before writing any handler mounting code. The generated REST endpoints use entrest's own query parameter format (`field.eq=value`, `field.has=value`, pagination via `page`/`per_page`) which is distinct from PeeringDB's Django-style filters -- that translation is Phase 6's concern.

**Primary recommendation:** Add entrest extension to entc.go with `DefaultOperations: [OperationRead, OperationList]`, add minimal schema-level annotations to all 13 schemas, run codegen, mount the generated handler at `/rest/v1/` on the shared mux with CORS middleware, and verify with integration tests.

### Pattern 1: entrest Extension Alongside entgql
**What:** Add entrest as a second extension to the existing `entc.Generate()` call, producing both GraphQL and REST outputs from the same schema definitions.
**When to use:** Any ent project that needs multiple API surfaces from a single schema.
**Example:**
```go
// Source: entrest getting-started guide + existing entc.go pattern
import (
    "github.com/lrstanley/entrest"
)

func main() {
    gqlExt, err := entgql.NewExtension(
        entgql.WithSchemaGenerator(),
        entgql.WithSchemaPath("../graph/schema.graphqls"),
        entgql.WithWhereInputs(true),
        entgql.WithConfigPath("../graph/gqlgen.yml"),
        entgql.WithRelaySpec(true),
    )
    if err != nil {
        log.Fatalf("creating entgql extension: %v", err)
    }

    restExt, err := entrest.NewExtension(&entrest.Config{
        Handler: entrest.HandlerStdlib,
        DefaultOperations: []entrest.Operation{
            entrest.OperationRead,
            entrest.OperationList,
        },
    })
    if err != nil {
        log.Fatalf("creating entrest extension: %v", err)
    }

    opts := []entc.Option{
        entc.Extensions(gqlExt, restExt),
        entc.FeatureNames("sql/upsert"),
    }

    if err := entc.Generate("./schema", &gen.Config{}, opts...); err != nil {
        log.Fatalf("running ent codegen: %v", err)
    }
}
```

### Pattern 2: Minimal Schema Annotation (per D-05)
**What:** Add a single schema-level entrest annotation to each entity's Annotations() method. Let entrest auto-discover fields for filtering/sorting.
**When to use:** When entrest's defaults are acceptable and per-field control is not needed.
**Example:**
```go
// Source: entrest annotation reference
func (Network) Annotations() []schema.Annotation {
    return []schema.Annotation{
        // Existing GraphQL annotations
        entgql.RelayConnection(),
        entgql.QueryField(),
        // entrest: read-only operations (per D-04, D-05)
        entrest.WithIncludeOperations(entrest.OperationRead, entrest.OperationList),
    }
}
```

### Pattern 3: REST Handler Mounting with Prefix
**What:** Mount the entrest-generated handler at a path prefix on the shared mux, wrapped with CORS middleware.
**When to use:** Mounting generated REST alongside other handlers on the same server.
**Example:**
```go
// Source: entrest kitchensink example + project main.go pattern
restSrv, err := rest.NewServer(entClient, &rest.ServerConfig{})
if err != nil {
    logger.Error("failed to create REST server", slog.String("error", err.Error()))
    os.Exit(1)
}

// Mount REST handler at /rest/v1/ prefix (per D-01)
restHandler := http.StripPrefix("/rest/v1", restSrv.Handler())
// Wrap with CORS middleware (per D-15)
restCORS := middleware.CORS(middleware.CORSInput{AllowedOrigins: cfg.CORSOrigins})
mux.Handle("/rest/v1/", restCORS(restHandler))
```

### Pattern 4: Edge Eager-Loading Annotations (per D-06)
**What:** Annotate edges to be eager-loadable via query parameters.
**When to use:** When API consumers should be able to request related data in a single response.
**Example:**
```go
// Source: entrest annotation reference
func (Organization) Edges() []ent.Edge {
    return []ent.Edge{
        edge.To("networks", Network.Type).
            Annotations(
                entrest.WithEagerLoad(true),
            ),
        edge.To("facilities", Facility.Type).
            Annotations(
                entrest.WithEagerLoad(true),
            ),
        // ... remaining edges
    }
}
```

### Anti-Patterns to Avoid
- **Per-field annotation overload (violates D-05):** Do NOT add `entrest.WithFilter()`, `entrest.WithSortable()` to every field individually. Let entrest auto-discover field types. Only add field-level annotations if the defaults are wrong.
- **Custom operation names/paths:** Do NOT use `WithOperationID()` or try to customize URL paths to match PeeringDB (that is Phase 6's concern). Let entrest use its defaults (D-02).
- **Wrapping entrest output for PeeringDB format:** Do NOT add middleware to transform entrest responses into PeeringDB envelope format. The two API surfaces are completely separate.
- **Adding write operations "just in case":** The `DefaultOperations` whitelist (D-04) is the sole enforcement. Do not add `WithExcludeOperations()` as backup -- it creates confusion about which is authoritative.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| REST endpoint generation | Custom HTTP handlers for each of 13 types | entrest codegen | 13 types x (list + read + pagination + filtering + sorting) = 130+ handler methods. entrest generates all of this. |
| OpenAPI spec | Manual OpenAPI YAML/JSON | entrest generated openapi.json | Keeping spec in sync with code is error-prone. entrest generates spec from the same source as handlers. |
| Pagination logic | Custom page/offset calculation | entrest PagedResponse | Pagination edge cases (last page, empty results, total count) are handled correctly by entrest. |
| Query parameter parsing for filtering | Custom filter parser for REST | entrest generated filter params | entrest generates type-safe parameter structs with proper validation per entity. |
| Edge eager-loading query params | Custom include/expand parser | entrest edge loading | entrest handles recursive edge loading with configurable limits (EagerLoadLimit). |

**Key insight:** entrest exists specifically to solve the "13 entity types x 5 query concerns" combinatorial explosion. Hand-rolling this produces 100+ methods that must stay in sync with the ent schema.

## Common Pitfalls

### Pitfall 1: entrest + entgql Extension Conflict
**What goes wrong:** Both extensions process the same schema annotations. Unknown whether they produce conflicting generated files or template collisions.
**Why it happens:** entgql and entrest are independent extensions never designed to be aware of each other. They generate code from the same schema definitions.
**How to avoid:** Run `go generate ./ent` immediately after adding the entrest extension and annotations. Compile before writing any handler mounting code. If codegen fails, both extensions can be debugged independently since they produce separate output files.
**Warning signs:** Code generation errors. Compilation errors in `ent/` directory after generation. Import cycle errors.

### Pitfall 2: Write Handlers Generated on Read-Only Database
**What goes wrong:** entrest generates CRUD by default. Without explicit read-only configuration, POST/PUT/PATCH/DELETE endpoints appear. On LiteFS replicas, these return 500 errors (SQLite read-only). On the primary, they could corrupt synced data.
**Why it happens:** entrest is designed for full CRUD applications. A read-only mirror is unusual.
**How to avoid:** Set `DefaultOperations: []entrest.Operation{entrest.OperationRead, entrest.OperationList}` in the Config (D-04). Verify the generated OpenAPI spec contains NO POST/PUT/PATCH/DELETE operations. Add a CI check parsing openapi.json for write operations.
**Warning signs:** OpenAPI spec listing non-GET methods. POST/PUT/PATCH returning 500 instead of 405.

### Pitfall 3: OpenAPI Spec Missing JSON Field Schemas
**What goes wrong:** Fields declared as `field.JSON("social_media", []SocialMedia{})` produce opaque `object` types in the OpenAPI spec because entrest cannot infer the schema from Go struct types automatically.
**Why it happens:** JSON fields in ent are stored as arbitrary JSON. entrest needs explicit `WithSchema()` annotations to describe their OpenAPI structure.
**How to avoid:** Per D-08, JSON fields are included as-is. The OpenAPI spec will describe them as generic objects unless `WithSchema()` is added. Since D-05 says minimal annotations, accept the generic schema for now. If OpenAPI consumers need precise types, add `WithSchema()` to `social_media`, `info_types`, and `available_voltage_services` fields.
**Warning signs:** OpenAPI spec showing `type: object` for fields that are actually typed arrays/objects.

### Pitfall 4: Path Prefix StripPrefix Off-By-One
**What goes wrong:** `http.StripPrefix("/rest/v1", handler)` strips the prefix before the handler sees the request. If the entrest handler expects paths starting with `/` (e.g., `/networks`), but the stripped path is `/networks`, this works. But if the prefix pattern on the mux is `/rest/v1/` (with trailing slash), the mux matches `/rest/v1/networks` but StripPrefix strips to `/networks` -- which is correct. Missing the trailing slash causes the mux pattern `"/rest/v1/"` to match differently from `"/rest/v1"` (the latter is an exact match, the former is a prefix match).
**Why it happens:** Go's ServeMux uses trailing slash to distinguish exact matches from prefix matches.
**How to avoid:** Register the mux pattern as `/rest/v1/` (with trailing slash for prefix matching). Use `http.StripPrefix("/rest/v1", handler)` (without trailing slash, so the stripped path starts with `/`).
**Warning signs:** 404 errors on REST endpoints. Requests reaching the wrong handler. OpenAPI spec endpoint returning 404.

### Pitfall 5: entrest DefaultEagerLoad Loading Everything
**What goes wrong:** If `DefaultEagerLoad: true` is set in Config, every query automatically loads all edges, causing massive JOIN queries that are slow on large datasets.
**Why it happens:** PeeringDB has deeply interconnected entities. Organization -> Networks -> NetworkIxLans -> IxLans -> InternetExchanges. Default eager loading traverses all of these.
**How to avoid:** D-06 says "all edges eager-loadable" but D-07 says "entrest default pagination." The Config field `DefaultEagerLoad` should remain `false` (the default). Instead, annotate each edge with `entrest.WithEagerLoad(true)` to make them AVAILABLE for eager loading, not loaded by default. Clients opt in via query parameters.
**Warning signs:** Extremely slow list queries. Large response bodies (MBs). Timeout errors on list endpoints.

### Pitfall 6: Readiness Middleware Not Covering /rest/ Paths
**What goes wrong:** The existing `readinessMiddleware` exempts specific paths (`/sync`, `/healthz`, `/readyz`, `/`). It already gates all other paths with 503 until first sync. Since `/rest/v1/` is not in the exemption list, it is automatically gated -- no code change needed.
**Why it happens:** The middleware uses a deny-by-default approach (gate everything, exempt specific infrastructure paths).
**How to avoid:** Verify the middleware works for REST paths in integration tests. Do NOT add `/rest/v1/` to the exemption list.
**Warning signs:** REST endpoints serving data before first sync completes.

## Open Questions

1. **entrest + entgql codegen coexistence**
   - What we know: Both extensions use separate annotation namespaces and generate separate output files. Both are registered via `entc.Extensions()`.
   - What's unclear: Whether they conflict at the template level or produce incompatible generated code. No documentation confirms they work together.
   - Recommendation: Run `go generate` as the FIRST task. If it fails, debug by running each extension in isolation. This is a blocking question -- all other work depends on it.

2. **entrest pagination response format vs D-10 (nullable fields as null)**
   - What we know: entrest generates `PagedResponse[T]` with `page`, `total_count`, `last_page`, `is_last_page`, `content` fields. Individual entities use ent's JSON serialization.
   - What's unclear: Whether ent's JSON serialization omits null fields or includes them as `null`. The `json:",omitempty"` behavior on generated ent structs needs verification.
   - Recommendation: After codegen, inspect the generated struct JSON tags. If ent uses `omitempty` on optional fields, evaluate whether `DisablePatchJSONTag: true` in entrest Config suppresses this behavior.

3. **entrest edge endpoint generation vs read-only**
   - What we know: entrest generates dedicated edge endpoints (e.g., `GET /networks/{id}/pocs`) in addition to eager-loading on the parent. Per GitHub issue #106, `WithIncludeOperations` may unexpectedly disable edge endpoints.
   - What's unclear: Whether `OperationRead + OperationList` at schema level preserves edge endpoints, or if edge endpoints are treated as separate operations.
   - Recommendation: After codegen, verify the generated OpenAPI spec includes edge endpoints. If missing, add `entrest.WithEdgeEndpoint(true)` to edge annotations.

4. **CORS middleware instance placement for REST**
   - What we know: D-15 says separate CORS middleware instance for REST. The existing middleware stack applies CORS globally (wraps the entire mux). A separate instance would need to wrap only REST routes.
   - What's unclear: Whether to apply CORS at the mux.Handle level (wrapping the REST handler specifically) or restructure the global middleware stack.
   - Recommendation: Wrap the REST handler with its own CORS middleware at mount time: `mux.Handle("/rest/v1/", restCORS(restHandler))`. The global CORS middleware will also fire, but since both use the same config (D-15), this is harmless -- the inner CORS adds headers and the outer CORS sees them and passes through.
