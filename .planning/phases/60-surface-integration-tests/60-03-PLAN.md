<objective>
Verify VIS-07: the anonymous shape of our `/api/{type}` response matches the committed upstream anonymous
fixture for every one of the 13 PeeringDB types, per D-08. This is the "absent-not-redacted" contract —
a Users-tier row must be structurally invisible in an anonymous response, not present-with-null-PII.

Phase 57's fixture capture (`testdata/visibility-baseline/beta/anon/api/{type}/page-1.json`) is the
assertion corpus. We replay each fixture against an httptest.Server wrapped around the full production
middleware chain + a DB seeded with the canonical seed.Full fixture, and assert `conformance.CompareResponses`
returns zero differences.

Why this lives in `internal/pdbcompat/`: the fixture-replay test targets the pdbcompat surface (`/api/`)
specifically, and the existing `golden_test.go` already demonstrates the httptest-against-Handler
pattern (`NewHandler(client).Register(mux)`). Reusing that file's layout keeps the test discoverable
for anyone touching pdbcompat.

Purpose: lock in the one architectural invariant that's hardest to spot by code review — "we never
leak a Users row even with all its PII nulled out" — by comparing to a real anonymous upstream capture.

Output: one new test file `internal/pdbcompat/anon_parity_test.go`.
</objective>

  <action>
    Create `internal/pdbcompat/anon_parity_test.go` implementing `TestAnonParityFixtures`, a table-
    driven test that replays each of the 13 committed anon fixtures against a local httptest.Server
    and asserts `conformance.CompareResponses` returns zero differences.

    Design decisions (locked):

    1. **Middleware depth: pdbcompat handler + privacy-tier middleware only**, not the full production
       chain. Rationale: the privacy-tier middleware is the only middleware that materially affects
       the response shape for this test; OTel, CORS, Gzip, Recovery, etc. are response-shape-neutral
       for a JSON envelope. Wiring the full chain would pull main-package imports into internal/pdbcompat,
       creating an import cycle. Instead we stamp `privctx.TierPublic` on the request ctx directly
       via a tiny inline middleware (equivalent to what PrivacyTier middleware does — but without the
       dependency on the main package's chainConfig).

       Concrete wiring:
       ```go
       mux := http.NewServeMux()
       pdbcompat.NewHandler(client).Register(mux)
       // Anonymous tier stamp — identical to the production privacyTier middleware effect.
       var handler http.Handler = http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
           ctx := privctx.WithTier(r.Context(), privctx.TierPublic)
           mux.ServeHTTP(w, r.WithContext(ctx))
       })
       srv := httptest.NewServer(handler)
       t.Cleanup(srv.Close)
       ```

    2. **DB seed: seed.Full**, not golden_test.go's bespoke 13-entity setup. Rationale: the assertion
       target is "Users rows filtered out on the anonymous surface" — seed.Full is the only fixture
       that actually contains Users rows (per Plan 01). The golden_test.go fixture is Public-only
       and wouldn't exercise the filter.

       ```go
       client := testutil.SetupClient(t)
       _ = seed.Full(t, client)  // populates all 13 types including 1 Public + 2 Users POCs
       ```

    3. **Fixture discovery: enumerate the 13 subdirectories of testdata/visibility-baseline/beta/anon/api/**
       via `os.ReadDir`. The resulting type names must exactly match the set expected by pdbcompat's
       route registry: `campus, carrier, carrierfac, fac, ix, ixfac, ixlan, ixpfx, net, netfac,
       netixlan, org, poc`. (This set is canonical in `internal/conformance/live_test.go allTypes`.)

       Relative path: tests in `internal/pdbcompat/` resolve the fixture at
       `../../testdata/visibility-baseline/beta/anon/api/{type}/page-1.json`. Derive the path with
       `filepath.Join` and log it on failure for reviewer debuggability.

    4. **Comparison strategy**:
       - Fetch `GET {srv.URL}/api/{type}?limit=1` (match upstream's typical pagination — the fixture
         is page-1 with the server's default page size, but our local DB has far fewer rows; using
         `?limit=1` gives both sides exactly one representative row for structural comparison).
       - Read reference bytes from the committed fixture.
       - Call `conformance.CompareResponses(refBody, ourBody)`.
       - On any Difference, `t.Errorf` with the Path, Kind, Details. Use `t.Run(typeName, ...)` so
         failures isolate per type.

       Why `?limit=1` rather than the full fixture: the VIS-01 corpus is upstream's production DB
       with thousands of rows per type. `CompareResponses` already compares `array[0]` elements as
       the element-shape probe (see `compareValues` in compare.go line 94-101), so a 1-row local
       response is sufficient to characterize the envelope and per-row structure.

    5. **Expected empty-diff types: 12 of 13**. The POC type is special: our anonymous response MUST
       NOT contain the seeded Users POCs (9000, 9001). We verify this as an additional assertion on
       the POC sub-test:
       ```go
       if typeName == "poc" {
           var env struct{ Data []map[string]any `json:"data"` }
           _ = json.Unmarshal(ourBody, &env)
           for _, row := range env.Data {
               if id, ok := row["id"].(float64); ok && (int(id) == 9000 || int(id) == 9001) {
                   t.Errorf("Users-tier POC id=%v leaked into anon /api/poc response", id)
               }
               if vis, ok := row["visible"].(string); ok && vis == "Users" {
                   t.Errorf("visible=Users row leaked into anon /api/poc response: id=%v", row["id"])
               }
           }
       }
       ```
       This is the "absent-not-redacted" invariant in executable form (VIS-07).

    6. **Structural tolerance for extra fields**: the fixture was captured live and may include fields
       that our pdbcompat layer doesn't emit (or vice versa). If `CompareResponses` returns any
       `extra_field` or `missing_field` Differences, the test fails — do NOT pre-filter them. The
       point is to catch shape drift. If specific fields are known-divergent (e.g. PeeringDB returns
       `meta.generated` and we don't yet), document them with a t.Log and XFAIL the specific Path(s)
