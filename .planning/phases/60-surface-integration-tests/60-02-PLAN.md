<objective>
Prove the ent Privacy policy fires through each of the 5 read surfaces (`/ui/`, `/graphql`, `/rest/v1/`,
`/api/`, `/peeringdb.v1.*`) for an anonymous caller, asserting (a) row counts in list responses exactly
match the Public-only count from seed.Full, and (b) detail lookups of the Users-tier IDs return each
surface's native not-found idiom (D-13/D-14).

This is complementary to Phase 59's `TestE2E_AnonymousCannotSeeUsersPoc`, which asserts "row absent" on
an ephemeral 1-POC fixture. Phase 60's version asserts list-count shapes against the canonical
seed.Full (3 POCs, 1 Public + 2 Users) — catching regressions where the filter passes scalar containment
checks but drops the wrong count (off-by-one, duplicate, mis-filtered parent row, etc.).

Purpose: regression-test the "list count" shape and the "detail 404 not 403" contract on the real
fixture used across the rest of the test suite.

Output: one new test file `cmd/peeringdb-plus/privacy_surfaces_test.go` with a single top-level
`TestPrivacySurfaces` using table-driven sub-tests per (surface, scenario) tuple.
</objective>

    3. `TestPrivacySurfaces` runs one sub-test per surface, each asserting:
       - LIST: the response for the canonical list endpoint has **exactly 1 POC row** (r.Poc, ID=500),
         and **zero rows with ID in {9000, 9001}**.
       - DETAIL: fetching the Users-tier ID returns the surface-native not-found idiom.

       Concrete sub-tests (using `t.Run` with `t.Parallel()` where safe):

       a. `pdbcompat_list_count` — `GET /api/poc` returns 200; decode envelope `{"data": [...]}`;
          assert `len(data) == 1` AND `data[0]["id"].(float64) == 500` AND every row satisfies
          `row["visible"].(string) == "Public"`. Failure message must include the full body for
          debuggability.

       b. `pdbcompat_detail_404` — `GET /api/poc/9000` returns 404. (Phase 59's E2E already covers
          this on its 1-POC fixture; repeat on the seed.Full corpus to prove it works when the table
          has more than one row — catches a failure mode where filter logic evaluates row-by-row but
          returns the first Public row instead of 404'ing on the Users-tier ID.)

       c. `rest_list_count` — `GET /rest/v1/pocs` returns 200; decode envelope `{"content": [...]}`;
          assert `len(content) == 1` AND `content[0]["id"].(float64) == 500`.

       d. `rest_detail_404` — `GET /rest/v1/pocs/9000` returns 404.

       e. `graphql_list_count` — `POST /graphql` with body
          `{"query":"{ pocs(first: 10) { edges { node { id name visible } } totalCount } }"}`;
          decode `data.pocs.totalCount` — must equal 1; `data.pocs.edges[*].node.id` — must contain
          only "500" (ids are strings in the GQL schema, cast accordingly). No node may have
          `visible == "Users"`.

       f. `grpc_list_count` — use `peeringdbv1connect.NewPocServiceClient(http.DefaultClient, fix.server.URL)`;
          call `ListPocs(ctx, connect.NewRequest(&pbv1.ListPocsRequest{}))`; assert
          `len(resp.Msg.GetPocs()) == 1` AND `resp.Msg.GetPocs()[0].GetId() == 500`.

       g. `grpc_detail_notfound` — same client; call `GetPoc(ctx, connect.NewRequest(&pbv1.GetPocRequest{Id: 9000}))`;
          assert `err != nil`; use `connect.CodeOf(err) == connect.CodeNotFound`. Per D-13/D-14 this
          MUST be CodeNotFound — a 403-equivalent (PermissionDenied) would leak existence.

       h. `ui_network_detail_no_leak` — `GET /ui/asn/13335` with `User-Agent: Mozilla/5.0` and
          `Accept: text/html,application/json` (matches e2e_privacy_test.go mustGet). Status must be
          200. The rendered HTML body MUST NOT contain the string "Users-Tier NOC" (r.UsersPoc.Name)
          and MUST NOT contain "users-noc@example.invalid" (r.UsersPoc.Email). It MAY contain
          "NOC Contact" (r.Poc.Name) — asserting presence is optional (the /ui/ renderer may or may
          not surface POC rows on the ASN page; the strong contract is "no leak of gated fields").

    4. All sub-tests may use `t.Parallel()` on each `t.Run` — the httptest.Server is safe for
       concurrent requests. The top-level test runs sequentially relative to other packages because
