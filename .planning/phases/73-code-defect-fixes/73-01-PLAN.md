The Campus schema currently has an `Annotations()` method on `ent/schema/campus.go`
(lines 132-141) that returns 5 entgql/entrest annotations:
```go
func (Campus) Annotations() []schema.Annotation {
    return []schema.Annotation{
        entgql.RelayConnection(),
        entgql.QueryField(),
        entrest.WithIncludeOperations(entrest.OperationRead, entrest.OperationList),
        entrest.WithDefaultSort("updated"),
        entrest.WithDefaultOrder(entrest.OrderDesc),
    }
}
```

This file is regenerated by `cmd/pdb-schema-generate` from `schema/peeringdb.json`.
Editing it directly would be silently stripped on the next `go generate ./...`.

ent's codegen discovers schema methods via REFLECTION on the schema TYPE —
the `(Campus)` receiver. Multiple files in the same package can declare methods
on the same type, and ent merges them. The sibling-file pattern (`fold_mixin.go`,
`{type}_fold.go`, `pdb_allowlists.go`, `poc_policy.go`) is a load-bearing
project convention.

**However:** Go does not permit two `Annotations()` methods on the same type.
This means the executor must either:
  (a) **Move** the existing `Annotations()` method to the sibling file and ADD
      `entsql.Annotation{Table: "campuses"}` to the returned slice — but this
      breaks the codegen, because `cmd/pdb-schema-generate`'s template will
      emit a duplicate `Annotations()` on the next regeneration.
  (b) **Modify the generator** so it does NOT emit the `Annotations()` block
      for Campus, leaving it solely to the sibling. Adds per-entity special-casing
      to the generator.
  (c) **Modify the generator's template** so the emitted `Annotations()` for
      Campus also includes `entsql.Annotation{Table: "campuses"}`. Inverts the
      sibling-file premise (the annotation lives in the generator template).
  (d) **Keep the existing `Annotations()` in `campus.go`** and add a *separate*
      contributor — Mixin pattern. Mixins return `[]ent.Mixin{}` and each Mixin's
      `Annotations()` method is ADDITIVE to the schema's `Annotations()`. This
      is exactly how `fold_mixin.go` works for the `_fold` columns.

**Choose option (d).** The sibling file declares a `campusTableAnnotation`
mixin (or similar — name is the executor's call), the schema generator's
output is unchanged, and ent merges the mixin's annotations with the
schema's. This matches the existing `fold_mixin.go` precedent exactly and
preserves the sibling-file convention.
