    Create a `Handler` struct that holds `*ent.Client` and exposes HTTP handler methods. Use CS-5 input struct pattern:

    ```go
    // Handler serves PeeringDB-compatible API endpoints.
    type Handler struct {
        client *ent.Client
    }

    // NewHandler creates a Handler for PeeringDB-compatible API endpoints.
    func NewHandler(client *ent.Client) *Handler {
        return &Handler{client: client}
    }
    ```

    Implement three handler methods that return `http.HandlerFunc`:

    **1. Index handler** (per D-17 -- Claude's discretion: include a JSON index):
    `func (h *Handler) Index() http.HandlerFunc` -- returns JSON map of type name to endpoint path:
    ```json
    {"campus":{"list_endpoint":"/api/campus"},"carrier":{"list_endpoint":"/api/carrier"},...}
    ```

    **2. List handler** (per D-01, D-02, D-05, D-09, D-12, D-16):
    `func (h *Handler) List(typeName string) http.HandlerFunc`
    Steps:
    1. Look up `typeName` in Registry (404 if not found)
    2. Parse query params: `ParsePaginationParams(r.URL.Query())` for limit/skip
    3. Parse filters: `ParseFilters(r.URL.Query(), tc.Fields)` (400 on error per D-24)
    4. Parse since: `ParseSinceParam(r.URL.Query())` (400 on invalid per D-15)
    5. Build QueryOptions struct with all parsed values
    6. Call `tc.List(r.Context(), h.client, opts)` to get results
    7. Call `WriteResponse(w, results)` to send PeeringDB envelope

    **3. Detail handler** (per D-03, D-22 -- single object wrapped in array per Pitfall 7):
    `func (h *Handler) Detail(typeName string) http.HandlerFunc`
    Steps:
    1. Extract ID from URL path (last segment after /api/{type}/)
    2. Parse to int (400 if not valid integer)
    3. Look up typeName in Registry
    4. Parse depth param from query (default 0, only 0 and 2 supported per D-06, D-07)
    5. Call `tc.Get(r.Context(), h.client, id, depth)`
    6. If ent.IsNotFound(err), return 404 with PeeringDB error envelope (per D-22: `{"meta":{"error":"..."}, "data":[]}`)
    7. Wrap result in single-element slice: `WriteResponse(w, []any{result})`

    **4. Register method** to set up routes on a mux:
    `func (h *Handler) Register(mux *http.ServeMux)`
    For each type in Registry (per D-01, D-02):
    - Register `"GET /api/{type}"` with List handler (exact path, no trailing slash)
    - Register `"GET /api/{type}/{id}"` pattern -- use a combined handler that checks if `{id}` is empty (list) or numeric (detail)

    Actually, Go 1.22+ ServeMux path parameters work well here. Register:
    - `mux.HandleFunc("GET /api/{type}", h.listHandler)` -- matches /api/net (no trailing slash)
    - `mux.HandleFunc("GET /api/{type}/", h.listHandler)` -- matches /api/net/ (trailing slash, per D-02)
    - `mux.HandleFunc("GET /api/{type}/{id}", h.detailHandler)` -- matches /api/net/42
    - `mux.HandleFunc("GET /api/", h.Index())` -- index (trailing slash catches /api/)
    - `mux.HandleFunc("GET /api", h.Index())` -- index (no trailing slash)

    Where `h.listHandler` and `h.detailHandler` are internal methods that extract the `{type}` and `{id}` path values from `r.PathValue()`.

    Validate the type name against Registry keys -- return 404 for unknown types.

    1. Add import: `"github.com/dotwaffle/peeringdb-plus/internal/pdbcompat"`

    2. After the REST server creation block and before the mux setup, create the compat handler:
    ```go

    compatHandler := pdbcompat.NewHandler(entClient)
    ```

    3. After creating the mux, register compat routes. Create a sub-mux for the compat handler and wrap with separate CORS per D-28:
    ```go
    // Mount PeeringDB compatibility API at /api/ with separate CORS per D-28.
    compatMux := http.NewServeMux()
    compatHandler.Register(compatMux)
    compatCORS := middleware.CORS(middleware.CORSInput{AllowedOrigins: cfg.CORSOrigins})
    mux.Handle("/api/", compatCORS(compatMux))
    mux.Handle("/api", compatCORS(compatMux))
    logger.Info("PeeringDB compat API mounted", slog.String("prefix", "/api/"))
    ```

    Wait -- if Register puts routes on a mux starting with /api/, and we mount that mux under /api/ with Handle, we get double-prefixing. Instead, the handler's Register should use full paths (/api/net, etc.) and register directly on the main mux. Then wrap the individual handlers with CORS.

    Better approach: Have Register return an http.Handler (the sub-mux with all routes registered). Then mount at root:
    ```go
    compatMux := http.NewServeMux()
    compatHandler.Register(compatMux)
    compatCORS := middleware.CORS(middleware.CORSInput{AllowedOrigins: cfg.CORSOrigins})
    // All /api/ routes go through compat CORS
    mux.Handle("/api/", compatCORS(compatMux))
    mux.Handle("/api", compatCORS(compatMux))
    logger.Info("PeeringDB compat API mounted", slog.String("prefix", "/api/"))
    ```

    This works because compatMux internally has patterns like "GET /api/net", and the outer mux routes /api/ prefixed requests to it.

    4. The readiness middleware in main.go already gates all non-infrastructure paths with 503 per D-29. The /api/ prefix is NOT in the bypass list (`/sync`, `/healthz`, `/readyz`, `/`), so it's automatically gated. No changes needed for readiness.
