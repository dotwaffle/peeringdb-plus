## [Unreleased] â€” v1.16

### Breaking

- **Removed `PDBPLUS_INCLUDE_DELETED` environment variable.** Sync now
  always persists deleted rows as tombstones (soft-delete via
  `UPDATE ... SET status='deleted'`). During the v1.16 â†’ v1.17 grace
  period, setting this variable triggers a startup WARN and is ignored;
  v1.17 upgrades this to a fatal startup error. Remove it from your
  environment. See `docs/CONFIGURATION.md Â§ Removed in v1.16`.

  **One-time gap:** Rows hard-deleted by sync cycles BEFORE the v1.16
  upgrade are gone forever. `?status=deleted` and `?since=N` queries
  populate going forward from the first post-upgrade sync cycle.

### Added

- **pdbcompat status Ã— since matrix** matching upstream
  `peeringdb_server/rest.py:694-727`. List requests without `?since`
  return only `status=ok` (STATUS-01). List requests with `?since=N`
  admit `(ok, deleted)`, plus `pending` for campus (STATUS-03).
  Single-object GETs (`/api/<type>/<id>`) admit `(ok, pending)` for
  all 13 entity types (STATUS-02). Explicit `?status=deleted` on a
  list request without `?since` silently returns an empty set,
  matching the upstream unconditional `filter(status='ok')` on
  `rest.py:725` (STATUS-04).

- **pdbcompat `?limit=0` semantics** match upstream `rest.py:734-737`:
  an explicit `limit=0` returns all matching rows (LIMIT-01). The
  default-when-unset remains `250`. `?depth=` on list endpoints is
  silently ignored in Phase 68; Phase 71 will add list+depth support
  with the `API_DEPTH_ROW_LIMIT=250` cap (LIMIT-02 guardrail).

### Changed

- **Sync now soft-deletes** instead of hard-deleting. The 13
  `deleteStale*` functions were renamed to `markStaleDeleted*`; they
  run `UPDATE ... SET status='deleted', updated=<cycle_start>` per
  sync cycle. Tombstone garbage-collection policy is deferred to
  SEED-004 (planted 2026-04-19).

### Deprecated

- `PDBPLUS_INCLUDE_DELETED` (see Breaking above; removal complete
  in v1.17).

### Fixed

- `?limit=0` on pdbcompat list endpoints previously fell back to
  `DefaultLimit=250`. Now returns all rows up to any other filter,
  matching upstream behaviour.

### Soft-delete tombstones (Phase 68)

Sync uses soft-delete (`UPDATE ... SET status='deleted', updated=<cycleStart>`) rather than hard-delete across all 13 entity types. The 13 functions live in `internal/sync/delete.go` as `markStaleDeleted*` (renamed from `deleteStale*` in v1.16 Phase 68 D-02). `cycleStart` is the `start := time.Now()` timestamp captured at the top of `Worker.Sync` and plumbed through `syncStep.deleteFn`.

When adding a new entity type, write the soft-delete function using the established pattern:
```go
func markStaleDeletedFoos(ctx context.Context, tx *ent.Tx, remoteIDs []int, cycleStart time.Time) (int, error) {
    return deleteStaleChunked(ctx, remoteIDs, func(chunk []int) (int, error) {
        return tx.Foo.Update().
            Where(foo.IDNotIn(chunk...)).
            SetStatus("deleted").
            SetUpdated(cycleStart).
            Save(ctx)
    }, "foos")
}
```
