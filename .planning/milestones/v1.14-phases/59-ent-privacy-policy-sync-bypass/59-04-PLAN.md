Install the read-path privacy floor in the data layer. Enable `gen.FeaturePrivacy`
in `ent/entc.go`, add `Policy()` to the POC schema, and prove via unit tests that
TierPublic callers cannot see `visible="Users"` rows while TierUsers callers can.

`privacy.DecisionContext`. Per RESEARCH.md Pitfall 2, the predicate is
`poc.Or(poc.VisibleEQ("Public"), poc.VisibleIsNil())` to handle three-valued SQL logic safely.

Purpose: One policy, five surfaces — graphql, rest/v1, api, grpcserver, ui all
inherit correct filtering from a single enforcement point at the ent runtime.

Output: `entc.go` config edit + generated `ent/privacy/` package + `Policy()` on
POC schema + behaviour tests covering VIS-04's five test cases (filter Users,
admit Public, admit NULL, admit via TierUsers, edge traversal).

## Trust Boundaries

| Boundary | Description |

| application handler → ent query builder | The policy is the chokepoint where visibility is enforced for all 5 API surfaces |
| SQL NULL semantics | Three-valued logic in WHERE: `visible = 'Public'` is FALSE for NULL — handled with Or(VisibleIsNil) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-59-03 | Elevation of Privilege / Information Disclosure | Edge-traversal query (e.g. `network.QueryPocs()`) bypassing the policy | mitigate | `PocQueryRuleFunc` type-asserts on `*ent.PocQuery` [RESEARCH Pattern 3]; the generated adapter catches edge-traversal queries because they materialise as the same type. If a future ent version introduces a new internal query type, the adapter returns `Denyf("unexpected query type")` — fail-closed. |

The schema's `Default("Public")` also backstops this. Two independent safeguards. |
