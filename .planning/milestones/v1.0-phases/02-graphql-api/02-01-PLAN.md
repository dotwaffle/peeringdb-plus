    **Step 2: Create graph/custom.graphql** with custom queries per D-05, D-11, and offset/limit pagination per D-01.

    The custom.graphql must define:
    1. Scalar types for SyncStatus fields
    2. The SyncStatus type and its query
    3. The networkByAsn dedicated query
    4. Offset/limit argument extensions on all 13 connection queries

    Per D-01, offset/limit must be real arguments on connection queries, not just documentation around cursor pagination. The approach (per research Pattern 9 and Pitfall 8): add `offset: Int` and `limit: Int` arguments to each of the 13 connection queries via GraphQL `extend type Query`. When provided, the resolver uses `.Offset()` and `.Limit()` on the ent query directly. When both cursor args (first/after/before/last) AND offset/limit are provided, the resolver returns an error (mutual exclusivity per Pitfall 8).

    ```graphql
    scalar Time
    scalar Map

    type SyncStatus {
      lastSyncAt: Time
      durationMs: Int
      objectCounts: Map
      status: String!
      errorMessage: String
    }

    extend type Query {
      syncStatus: SyncStatus
      networkByAsn(asn: Int!): Network

      # Offset/limit pagination alternatives per D-01.
      # These provide simple offset-based pagination for scripts and non-Relay clients.
      # Mutually exclusive with cursor pagination (first/after/before/last).
      # limit defaults to 100, max 1000 per D-14. offset defaults to 0.
      organizationsList(offset: Int, limit: Int, where: OrganizationWhereInput): [Organization!]!
      networksList(offset: Int, limit: Int, where: NetworkWhereInput): [Network!]!
      facilitiesList(offset: Int, limit: Int, where: FacilityWhereInput): [Facility!]!
      internetExchangesList(offset: Int, limit: Int, where: InternetExchangeWhereInput): [InternetExchange!]!
      pointOfContactsList(offset: Int, limit: Int, where: PointOfContactWhereInput): [PointOfContact!]!
      ixLansList(offset: Int, limit: Int, where: IXLanWhereInput): [IXLan!]!
      ixPrefixesList(offset: Int, limit: Int, where: IXPrefixWhereInput): [IXPrefix!]!
      ixFacilitiesList(offset: Int, limit: Int, where: IXFacilityWhereInput): [IXFacility!]!
      networkIxLansList(offset: Int, limit: Int, where: NetworkIXLanWhereInput): [NetworkIXLan!]!
      networkFacilitiesList(offset: Int, limit: Int, where: NetworkFacilityWhereInput): [NetworkFacility!]!
      carriersList(offset: Int, limit: Int, where: CarrierWhereInput): [Carrier!]!
      carrierFacilitiesList(offset: Int, limit: Int, where: CarrierFacilityWhereInput): [CarrierFacility!]!
      campusesList(offset: Int, limit: Int, where: CampusWhereInput): [Campus!]!
    }
    ```

    Note: The offset/limit queries return plain arrays `[Type!]!` (not connection types). This is intentional -- offset/limit is a simpler pagination model that does not need edges/pageInfo/cursors. The WhereInput types are reused from the generated schema for filtering consistency. The exact WhereInput type names come from the generated schema.graphqls -- read it to confirm the names match (e.g., it might be `NetworkWhereInput` or `NetworkFilterInput`).

    The `Time` and `Map` scalars must be defined here for the SyncStatus type. The entity types (Network, Organization, etc.) and their WhereInput types are defined in the generated schema.graphqls.
