    Note: Do NOT include HEALTHCHECK in Dockerfile.prod -- Fly.io uses its own health check mechanism defined in fly.toml. Do NOT switch to non-root user -- LiteFS requires FUSE mount permissions per Pitfall 4.

    **Create litefs.yml per D-17, D-20, D-27:**

    ```yaml
    # LiteFS configuration for PeeringDB Plus
    # See: https://fly.io/docs/litefs/config/

    # FUSE mount point where the application accesses the database
    fuse:
      dir: "/litefs"

    # Internal storage directory for replication state (on the Fly.io volume)
    data:
      dir: "/var/lib/litefs"
      compress: true
      retention: "10m"

    # LiteFS launches the app as a subprocess per D-20
    exec:
      - cmd: "/usr/local/bin/peeringdb-plus"

    # Internal replication API (node-to-node communication)
    http:
      addr: ":20202"

    # HTTP proxy for write forwarding and read consistency
    proxy:
      addr: ":8080"
      target: "localhost:8081"
      db: "peeringdb-plus.db"
      passthrough:
        - "/healthz"
        - "/readyz"

    # Consul-based leader election per D-17, D-27
    lease:
      type: "consul"
      candidate: ${FLY_REGION == PRIMARY_REGION}
      advertise-url: "http://${FLY_ALLOC_ID}.vm.${FLY_APP_NAME}.internal:20202"
      consul:
        url: "${FLY_CONSUL_URL}"
        key: "${FLY_APP_NAME}/primary"
    ```

    Key design points:
    - `passthrough` includes /healthz and /readyz so health checks bypass LiteFS consistency logic per Pitfall 5
    - `proxy.addr: ":8080"` is what Fly.io routes to (external port)
    - `proxy.target: "localhost:8081"` is where the app listens (internal port)
    - `lease.candidate` only becomes true in the primary region
    - `exec` runs the app as a LiteFS subprocess per D-20

    # Fly.io deployment configuration for PeeringDB Plus

    - Health check on `/healthz` (liveness, always 200) with 30s grace for LiteFS mount + first sync
    - Volume mounted at `/var/lib/litefs` for LiteFS replication data per Pitfall 7
