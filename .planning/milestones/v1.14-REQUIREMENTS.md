## v1.14 Requirements

Requirements for the Authenticated Sync & Visibility Layer milestone. Each maps to roadmap phases 57-62.

### Visibility (VIS)

- [x] **VIS-01**: System captures both unauthenticated and authenticated PeeringDB API responses for all 13 types as committed baseline fixtures (`testdata/visibility-baseline/beta/` plus a confirmation pass against `www.peeringdb.com` for poc/org/net under `testdata/visibility-baseline/prod/`)
  - **Outcome:** Phase 57. Committed fixtures: `beta/anon/api/` (26 raw anon pages, 13 types × 2 pages), `beta/auth/api/` (26 redacted auth pages with `<auth-only:TYPE>` PII placeholders), `prod/anon/api/{poc,org,net}/` (6 anon-only prod pages). `TestCommittedFixturesHaveNoPII` PASS on committed tree.
- [x] **VIS-02**: System emits a structural diff report from the captured fixtures listing every field/row that differs between unauth and auth responses, with a per-type table that's reviewable in code review
  - **Outcome:** Phase 57. `visbaseline.Diff` + Markdown/JSON emitters produce `testdata/visibility-baseline/DIFF.md`, `DIFF-beta.md`, `DIFF-prod.md`, `diff.json`. 13 per-type tables with TOC for code review. Empirical finding: only two auth-gated surfaces exist across all 13 types.
- [x] **VIS-03**: ent schemas have visibility-bearing fields for every auth-gated entity identified by VIS-02 (`poc.visible` already exists; add others as the diff surfaces them and regenerate ent)
  - **Outcome:** Phase 58. No new ent fields required — `poc.visible` (pre-existing, row-level) and `ixlan.ixf_ixp_member_list_url_visible` (pre-existing, per-field) cover the two surfaces. `internal/visbaseline/schema_alignment_test.go` regression test locks the allowlist. `go generate ./...` drift-clean.
- [x] **VIS-04**: ent Privacy query policy filters non-`Public` rows from anonymous responses on POC and any other types from VIS-03; policy is loaded via `entgo.io/ent/privacy` and wired in `ent/entc.go`
  - **Outcome:** Phase 59. `gen.FeaturePrivacy` enabled in `ent/entc.go`; `ent/privacy/privacy.go` generated; `(Poc).Policy()` with NULL-safe predicate `poc.Or(VisibleEQ("Public"), VisibleIsNil())` treating NULL as schema-default `Public`. `TestPocPolicy_*` (5 tests) lock behaviour.
- [x] **VIS-05**: Sync worker bypasses the privacy policy via `privacy.DecisionContext(ctx, privacy.Allow)` so it can read/write the full dataset; tests assert the bypass is in effect for sync-context goroutines and absent everywhere else
  - **Outcome:** Phase 59. `worker.go:260` is the sole call site, placed as first statement of `Worker.Sync` before CAS and before OTel span start. `TestSyncBypass_SingleCallSite` is regex-audited (nested-paren tolerant, comment-aware) across `internal/ cmd/ ent/schema/ graph/` skipping `*_test.go`, generated `ent/`, and `internal/testutil/`. Child goroutines inherit via ctx parentage — locked by `TestWorkerSync_ChildGoroutineInheritsBypass`.
- [x] **VIS-06**: All 5 read surfaces (`/ui/`, `/graphql`, `/rest/v1/`, `/api/`, `/peeringdb.v1.*`) honour the privacy policy; per-surface integration tests assert no `visible=Users` row leaks on an anonymous request
  - **Outcome:** Phase 60. `TestPrivacySurfaces` (9 sub-tests list-count + detail-404) + `TestE2E_AnonymousCannotSeeUsersPoc` (19 sub-tests across 5 surfaces × 2 tiers) via real `buildMiddlewareChain`. `seed.Full` extended with `UsersPoc` (id=9000) + `UsersPoc2` (id=9001) + `AllPocs`.
- [x] **VIS-07**: pdbcompat `/api/poc` (and embedded `poc_set`) anonymous response shape matches upstream anonymous shape — rows absent, not redacted; verified by replaying the VIS-01 fixtures against our endpoint with an empty diff
  - **Outcome:** Phase 60. `TestAnonParityFixtures` (13 sub-tests fixture replay) asserts empty structural diff modulo one documented non-privacy divergence (`ixpfx.notes` — allow-listed in `knownDivergences`). Conformance live test flipped to anon-vs-anon only.

### Sync Configuration (SYNC)

- [x] **SYNC-01**: Authenticated sync becomes the recommended deployment configuration; documented in `docs/CONFIGURATION.md`/`docs/DEPLOYMENT.md` and the production Fly.io app has `PDBPLUS_PEERINGDB_API_KEY` set as a fly secret
  - **Outcome:** Phase 62. Fly.io app `peeringdb-plus` has `PDBPLUS_PEERINGDB_API_KEY` set; `fly secrets list` + `fly logs | grep 'sync mode'` (`auth=authenticated`) confirm rollout. `docs/DEPLOYMENT.md` Authenticated PeeringDB Sync (Recommended) section + `docs/CONFIGURATION.md` mark the env var as Recommended. 62-HUMAN-UAT.md 6/6 PASS.
- [x] **SYNC-02**: Unauthenticated sync remains a first-class supported configuration — omit the API key, worker syncs the upstream anonymous payload only, no `Users`-tier rows ever land in the DB; verified by an integration test
  - **Outcome:** Phase 60. `TestNoKeySync` Phase A: 0 Authorization headers on upstream, 0 `Users` rows in DB. Phase B: surface count == DB count with `TierPublic` ctx. Privacy filter is effectively a no-op on read.
- [x] **SYNC-03**: `PDBPLUS_PUBLIC_TIER` env var (default `public`, accepts `users`) elevates all anonymous callers to Users-tier for private-instance deployments; tests cover both values
  - **Outcome:** Phase 59. `config.parsePublicTier` strict switch; `Config.PublicTier` flows through `chainConfig.DefaultTier` → `PrivacyTier` middleware. Fails fast on invalid value. `TestE2E_PublicTierUsersAdmitsRow` + D-15 coverage.
- [x] **SYNC-04**: Startup logs a WARN line when `PDBPLUS_PUBLIC_TIER=users` is in effect, so the elevated default is never silent
  - **Outcome:** Phase 61. `logStartupClassification` emits conditional `slog.Warn("public tier override active", ...)`. `TestStartup_WarnsOnUsersTier` + `TestStartup_NoWarnOnPublicTier` lock behaviour.

## Deferred to v1.15 (or later)

### OAuth Identity (AUTH)

- **AUTH-01**: User can log in via PeeringDB OAuth (`auth.peeringdb.com`) using the authorization code flow with `profile`+`networks` scopes
- **AUTH-02**: An OAuth-identified caller's request context carries `tier=Users`, causing the ent Privacy policy to admit `Users`-visibility rows for that caller
- **AUTH-03**: Session/JWT mechanism plumbs OAuth identity from the OAuth callback through middleware into the ent context
- **AUTH-04**: `/about` shows the logged-in user (when authenticated) and which `networks` they administer per the OAuth `networks` scope

### Domain Extensions (Carried from v1.13 deferred list)

- **BGP-01**: Per-ASN BGP summary from bgp.tools (prefix counts, RPKI coverage)
- **IRR-01**: IRR/AS-SET membership from WHOIS source
- **PFX-01**: IP prefix lookup with origin ASN, RPKI status

### Operational Verification (Carried)

- **OPVR-01**: `fly_region` Grafana template variable verified against live multi-region deployment
- **OPVR-02**: Go runtime metric names verified against live Grafana Cloud
- **OPVR-03**: CI coverage pipeline verified on actual GitHub Actions run
- **OPVR-04**: v1.13 phase 52 CSP enforcement smoke test (Chrome devtools) and phase 53 header smoke tests (curl HSTS/XFO/XCTO, slowloris)

## Out of Scope

| Feature | Reason |
|---------|--------|
| Per-user PeeringDB API key issuance for downstream callers | Out of scope at the project level — server-side config only |
| OAuth in v1.14 | Deferred to v1.15 — privacy floor must ship first to avoid coupling concerns |
| Field-level redaction of non-`Public` rows in responses | We mirror upstream behaviour: rows are absent from anonymous responses, not present-with-redacted-fields |
| Two-way OAuth (we don't act as an OAuth provider) | Mirror is read-only; OAuth use is consumer-only against `auth.peeringdb.com` |
| Detecting whether an OAuth user has org membership for arbitrary records | PeeringDB OAuth `networks` scope is per-network only; we admit any OAuth-authenticated caller to Users-tier (matches what an authenticated PeeringDB API request would see) |
| Real-time visibility-change propagation | Visibility changes ride the existing sync interval — no push channel from upstream exists |
