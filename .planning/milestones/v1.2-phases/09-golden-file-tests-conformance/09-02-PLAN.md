**Create `internal/conformance/compare.go`:**

**Package documentation (per API-1):**
```go
// Package conformance provides structural comparison of JSON API responses
// for validating PeeringDB compatibility layer output against the real
// PeeringDB API. Comparison is structure-only: field names, value types,
// and nesting depth are checked, but actual values are not.
```

**`Difference` type (per CONTEXT.md decision: structure-only comparison):**
```go
// Difference describes a structural mismatch between two JSON responses.
type Difference struct {
    Path    string // JSON path, e.g., "data[0].net_set[0].asn"
    Kind    string // "missing_field", "extra_field", "type_mismatch"
    Details string // Human-readable description
}
```

**`CompareStructure` function:**
- Signature: `func CompareStructure(reference, actual map[string]any) []Difference`
- Per CONTEXT.md: compares field names, types, nesting -- not values
- Walk both maps recursively:
  - For each key in reference not in actual: add `missing_field` Difference
  - For each key in actual not in reference: add `extra_field` Difference
  - For matching keys: compare JSON types (string, number, bool, null, array, object)
  - For nested objects: recurse with path prefix like "parent.child"
  - For arrays: if both non-empty, compare structure of first element only (PeeringDB data arrays have homogeneous elements)
  - Sort differences by Path for deterministic output

**`jsonType` helper function:**
- Returns string type name for a value: "string", "number", "bool", "null", "array", "object"

**`ExtractStructure` function:**
- Signature: `func ExtractStructure(jsonBody []byte) (map[string]any, error)`
- Parses a PeeringDB JSON envelope `{"meta":{},"data":[...]}` and returns the structure
- Uses `json.Unmarshal` with `json.Number` decoder for preserving numeric types
- Wraps errors per ERR-1: `fmt.Errorf("unmarshal response: %w", err)`

**`CompareResponses` function (convenience wrapper):**
- Signature: `func CompareResponses(reference, actual []byte) ([]Difference, error)`
- Calls `ExtractStructure` on both inputs, then `CompareStructure`
- Returns combined errors if either extraction fails

**Create `internal/conformance/compare_test.go`:**
- Table-driven tests (per T-1) for CompareStructure:
  - Test: identical flat structures -> no differences
  - Test: missing field -> one missing_field difference
  - Test: extra field -> one extra_field difference
  - Test: type mismatch (string vs number) -> one type_mismatch difference
  - Test: nested object with missing inner field -> path shows "outer.inner"
  - Test: array with different element structure -> detects structural difference
  - Test: null vs object -> type_mismatch
  - Test: empty arrays on both sides -> no differences
- Tests for ExtractStructure:
  - Valid PeeringDB envelope -> extracts structure successfully
  - Invalid JSON -> returns wrapped error
- Tests for CompareResponses:
  - Two matching responses -> no differences
  - Structurally different responses -> reports differences
- All tests use `t.Parallel()` per T-3

**Create `cmd/pdbcompat-check/main.go` (per CONTEXT.md: standalone binary at cmd/pdbcompat-check/):**

The CLI tool does the following per CONTEXT.md decision:
1. Fetches a response from beta.peeringdb.com for a given type (e.g., `GET https://beta.peeringdb.com/api/net?limit=1`)
2. Creates entities locally from that data, renders through our serializer
3. Compares the structural output

**Implementation:**

```go
package main

import (
    "context"
    "encoding/json"
    "flag"
    "fmt"
    "io"
    "log/slog"
    "net/http"
    "os"
    "sort"
    "time"

    "github.com/dotwaffle/peeringdb-plus/internal/conformance"
)
```

**Flags:**
- `-url` (default: `"https://beta.peeringdb.com"`): PeeringDB API base URL (per CONTEXT.md: uses beta, not production)
- `-type` (default: `""`): PeeringDB type to check (empty = all 13 types)
- `-timeout` (default: `30s`): HTTP request timeout (per SEC-1: explicit I/O timeouts)

**Main logic:**
- Parse flags
- Create HTTP client with explicit timeout (per SEC-1)
- Determine types to check: if `-type` is set, check that one; otherwise iterate all 13 types from a hardcoded sorted list: `[]string{"campus", "carrier", "carrierfac", "fac", "ix", "ixfac", "ixlan", "ixpfx", "net", "netfac", "netixlan", "org", "poc"}`
- For each type:
  1. Fetch `GET {baseURL}/api/{type}?limit=1` with the HTTP client
  2. Read response body, verify status 200
  3. Call `conformance.ExtractStructure(body)` on the PeeringDB response
  4. Create a local in-memory "expected" structure by defining what our compat layer would return for the same type (hardcode the known envelope structure `{"meta":{},"data":[{...}]}`)
  5. Actually, simpler approach per CONTEXT.md: just compare the PeeringDB response structure against itself -- the tool's PRIMARY purpose is to detect if PeeringDB's structure has CHANGED from what we expect. So compare the fetched structure against a known-good structure from our golden files or a hardcoded expectation.
  6. REVISED approach (matching CONTEXT.md exactly): "fetch from beta.peeringdb.com, create entities locally from that data, render through our serializer, compare":
     - Fetch raw JSON from beta.peeringdb.com
     - Extract the structure of PeeringDB's response
     - Also extract the structure of a sample golden file for that type (or if no golden file, just report the PeeringDB structure)
     - Use `conformance.CompareStructure` to find structural differences
     - BUT: since we cannot easily create entities from PeeringDB data and render through our serializer in a standalone CLI (it requires an ent client with database), simplify to: compare PeeringDB response structure against our golden file structure for the same type

  7. FINAL approach (pragmatic for CLI):
     - Fetch from beta.peeringdb.com
     - Read corresponding golden file from `internal/pdbcompat/testdata/golden/{type}/list.json`
     - Compare structures using `conformance.CompareResponses`
     - Report differences

- Print results per type: "OK" or list of Difference items
- Exit with code 0 if all match, code 1 if any differences found
- Use structured output (per OBS-1): `slog.Info("checking type", slog.String("type", typeName))`

**Rate limiting:** Add a `time.Sleep(3 * time.Second)` between type fetches to respect PeeringDB rate limits (per Pitfall 6). Or better, use a simple sequential loop with the sleep to avoid needing the full peeringdb.Client.

**Error handling (per ERR-1):** Wrap all errors with context. If a type fetch fails, log error and continue to next type. Report summary at end.
