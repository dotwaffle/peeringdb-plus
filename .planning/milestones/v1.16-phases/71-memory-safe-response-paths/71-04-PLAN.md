Wire Plans 01 + 03 into `internal/pdbcompat/handler.go`'s `serveList`. Replace the current `json.NewEncoder(w).Encode(envelope{Meta, Data})` materialisation with:

1. Pre-flight SELECT COUNT(*) via the new `tc.Count` closure (Task 1 below).
2. `CheckBudget(count, tc.Name, 0, h.responseMemoryLimit)` — depth is always 0 on list paths per Phase 68 LIMIT-02 guardrail.
3. If over budget → `WriteBudgetProblem` + structured log + return.
4. If under budget → call existing `tc.List` to materialise rows → drive `StreamListResponse` from a slice-backed iterator.

**Why stream from an already-materialised slice?**

- Budget is the teeth; streaming is the hygiene. The pre-flight 413 is what actually prevents OOM on over-budget requests. Streaming within the under-budget bucket still improves tail latency (time-to-first-byte) and gives the GC breathing room to reclaim per-row allocations between flushes.
- Incremental refactor path. A future plan can flip the 13 closures to cursor-based iterators without touching the handler contract this plan lands.

Output: Edits to handler.go + registry.go + registry_funcs.go + main.go + new stream_integration_test.go. Existing tests must still pass; new tests cover 413 + streamed 200 + byte-exact parity with legacy envelope.
