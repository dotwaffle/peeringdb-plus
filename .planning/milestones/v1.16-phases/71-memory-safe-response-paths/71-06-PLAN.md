    ## Response Memory Envelope

    v1.16 Phase 71 — pdbcompat list responses are gated by a per-request
    memory budget so the 256 MB Fly replicas never OOM under `limit=0`,
    depth=2, or 2-hop traversal responses enabled by Phases 68 + 70.

    ### The envelope

    256 MB replica total
      − 80 MB Go runtime baseline (observed v1.15)
      − 48 MB slack (other in-flight requests + GC overhead)
      = 128 MiB PDBPLUS_RESPONSE_MEMORY_LIMIT default

    Operators tune via the env var; 0 disables the check (dev only).

    ### The three moving parts

    | File | Responsibility |
    |---|---|
    | `internal/pdbcompat/stream.go` | Hand-rolled JSON token writer. `StreamListResponse(w, meta, rowsIter)` emits `{"meta":...,"data":[...]}` with per-row `json.Marshal` and periodic `http.Flusher.Flush()` (every 100 rows). No full-slice materialisation on the wire. |
    | `internal/pdbcompat/rowsize.go` | Hardcoded `map[string]RowSize{Depth0, Depth2}` calibrated from `bench_row_size_test.go` then doubled (D-03). Conservative by design — false-positive 413s are preferred over OOM. Recalibrated every major milestone. |
    | `internal/pdbcompat/budget.go` | `CheckBudget(count, entity, depth, budgetBytes)` returns `(BudgetExceeded, bool)`. Over-budget requests get RFC 9457 `application/problem+json` 413 via `WriteBudgetProblem` BEFORE any row data is fetched. |

    ### Per-entity worst-case sizing

    Values are the DOUBLED figure from `BenchmarkRowSize_*` (Plan 02
    calibration). At the 128 MiB default budget, the max unfiltered
    list row count is `128 MiB / typicalRowBytes(entity, depth=0)` —
    for networks this is roughly {max_rows_placeholder} rows; for
    ix_prefixes {max_rows_placeholder}; etc. Exact values: see the
    `typicalRowBytes` map in `internal/pdbcompat/rowsize.go`.

    | Entity | depth=0 | depth=2 |
    |---|---|---|
    | org | {fill from Plan 02 calibration} | {fill} |
    | net | {fill} | {fill} |
    | fac | {fill} | {fill} |
    | ix  | {fill} | {fill} |
    | ... | ... | ... |

    (Full table in `internal/pdbcompat/rowsize.go`.)

    ### Request lifecycle

    1. Client sends `GET /api/<type>?filters...&limit=0`
    2. Handler parses filters + since + pagination (unchanged from v1.6)
    3. **Pre-flight:** handler runs `tc.Count(ctx, client, opts)` — a
       filtered `SELECT COUNT(*)` using the same predicate chain as the
       upcoming `tc.List` call
    4. **Budget check:** `CheckBudget(count, tc.Name, 0, cfg.ResponseMemoryLimit)`
       - under budget → step 5
       - over budget → 413 `application/problem+json` with `max_rows`,
         `budget_bytes`, `detail` — NO row data is fetched
    5. `tc.List` materialises the result slice
    6. `StreamListResponse` emits the envelope token-by-token with
       `http.Flusher.Flush()` every 100 rows

    ### Telemetry (MEMORY-03)

    - OTel span attribute `pdbplus.response.heap_delta_kib` — per-request
      `runtime.MemStats.HeapInuse` delta, sampled once at entry and once
      at exit (STW ~µs; NEVER per row)
    - Prometheus histogram `pdbplus_response_heap_delta_kib{endpoint,entity}`
      — p50/p95/p99 visible in Grafana's "Response Heap Delta (Phase 71)"
      panel (SEED-001 watch row)
    - When `PDBPLUS_RESPONSE_MEMORY_LIMIT > 0` is breached, handler emits
      `slog.Warn("pdbcompat: response budget exceeded", ...)` before
      writing the 413

    ### Out of scope

    Streaming + budget apply to pdbcompat ONLY. grpcserver already
    streams via keyset pagination; entrest uses ent-generated handlers
    that don't buffer unbounded; GraphQL has its own depth/complexity
    limits (v1.12). See Phase 71 CONTEXT.md D-07 for the full scope
    rationale.

    ### Extending

    Adding a new entity type requires:
    1. `internal/pdbcompat/rowsize.go` — add an entry to `typicalRowBytes`
       (run `BenchmarkRowSize` against a seeded fixture, double the mean)
    2. `internal/pdbcompat/registry_funcs.go` — add a new `wireFooFuncs`
       function that pairs a List closure with a Count closure via a
       shared `fooPredicates` local helper (preserves Phase 68/69
       predicate invariants)
    3. New row in this table
