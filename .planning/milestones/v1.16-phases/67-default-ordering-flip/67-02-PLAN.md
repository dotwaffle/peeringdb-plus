<objective>
Create a project-local override of entrest's `sorting.tmpl` template under `ent/templates/entrest-sorting/` that injects compound `(-updated, -created, -id)` tie-breakers into the generated `applySorting<Type>` helpers when the requested sort field matches the schema's declared default sort (which is `"updated"` after Plan 01). Wire `entc.TemplateDir` in `ent/entc.go` and regenerate `ent/rest/sorting.go`.

Purpose: Per D-02 (updated 2026-04-19) and D-07 (added 2026-04-19), entrest's annotation API cannot express compound default order â€” `WithDefaultOrder` accepts a single `SortOrder` enum, and the wire protocol is single-field `?sort=<f>&order=<dir>`. Plan 01 gives the declarative single-field default (`updated DESC`). This plan injects the `created DESC, id DESC` tie-breakers into the generated `applySorting<Type>` bodies so the actual emitted SQL matches pdbcompat and grpcserver's compound order.

Per D-04 clarification (2026-04-19): entrest's eager-load template also calls `applySorting<Type>` on nested `_set` arrays at `depth>=1`. The template override will therefore also reorder nested arrays â€” this is accepted as correct behaviour (see D-04 clarification; upstream PeeringDB behaves the same way). Plan 06's cross-surface E2E asserts this.

Output: in-tree template override, wired generator, regenerated `ent/rest/sorting.go` with compound tie-break injection in all 13 `applySorting<Type>` helpers.
</objective>
