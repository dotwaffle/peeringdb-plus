    ```markdown
    ## Ordering

    All list endpoints return rows in compound `(-updated, -created, -id)` order by default, matching the upstream PeeringDB `django-handleref` base `Meta.ordering = ("-updated", "-created")` plus a deterministic `id DESC` tertiary tiebreaker for cross-replica consistency.

    - **pdbcompat** (`/api/<type>`) and **grpcserver** (`List*`/`Stream*` RPCs) emit the full compound ORDER BY directly via ent.
    - **entrest** (`/rest/v1/<type>`) uses an in-tree `ent/templates/entrest-sorting/sorting.tmpl` override because entrest's annotation API is single-field; the template injects `created, id` tie-breakers (in the requested sort direction) when the default sort path is taken. Explicit `?sort=<field>&order=<dir>` overrides are honoured unchanged.
    - **Nested `_set` arrays at depth>=1 (entrest only):** entrest's eager-load template calls `applySorting<Type>` on nested relations, so `/rest/v1/<type>?depth>=1` responses also adopt the compound order. This matches upstream PeeringDB's Django serializer behaviour.
    - **grpcserver streaming** uses a compound `(lastUpdated, lastID)` keyset cursor (base64-encoded as `RFC3339Nano:id`) for stable pagination under concurrent mutation. Wire proto type is `string page_token` (opaque); no consumer-facing changes.
    - **Performance:** all 13 entity tables carry an `updated` index (added Phase 67 per [ORDER-01..03](../.planning/REQUIREMENTS.md)); `ORDER BY updated DESC, id DESC` hits an index scan instead of a full-table sort.
    ```
