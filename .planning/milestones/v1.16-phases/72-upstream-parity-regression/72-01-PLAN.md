Create `cmd/pdb-fixture-port/main.go`.

Structure (keep under ~300 LOC in main.go; split parse/render into helpers within the same file or sibling files if it grows):

Flag surface (stdlib `flag` — match cmd/pdb-compat-allowlist/main.go style):
  --upstream-file string    local path to pdb_api_test.py (default empty)
  --upstream-ref string     git ref to fetch via `gh api` if --upstream-file absent (default "main")
  --out string              output file path (default "internal/testutil/parity/fixtures.go")
  --category string         which fixture category to emit ("ordering" for this plan)
  --check                   advisory drift-check mode; does not write
  --pinned string           expected upstream SHA (used by --check)

Main flow:
  1. Parse flags; validate mutually-exclusive (--upstream-file vs --upstream-ref).
  2. Resolve upstream bytes: if --upstream-file set → read file; else → exec `gh api repos/peeringdb/peeringdb/contents/src/peeringdb_server/management/commands/pdb_api_test.py --jq .content | base64 -d` (use os/exec with a composed pipeline). Write bytes to a temp file via os.CreateTemp for hashing. Record resolved ref via `gh api repos/peeringdb/peeringdb/commits/<ref>` → commit SHA.
  3. Compute sha256 of upstream bytes → `upstreamHash`.
  4. If `--check` mode: compare upstreamHash to `--pinned` (or to hash embedded in current --out file header if --pinned empty); exit 0 on match, exit 1 on drift with stderr message.
  5. Otherwise: parse the upstream file per --category (this plan: "ordering"), extract fixtures, render to temp file, os.Rename onto --out.

Rendering:
  - Header comment block:
      // Code generated by cmd/pdb-fixture-port — DO NOT EDIT.
      // Upstream:     peeringdb/peeringdb@<commitSHA>
      // UpstreamHash: sha256:<hex>
      // Source:       src/peeringdb_server/management/commands/pdb_api_test.py
      // Category:     ordering
      // Ported:       <YYYY-MM-DD>
  - Package clause: `package parity`
  - `type Fixture struct {...}` (defined in the first render; subsequent renders overwriting this file MUST produce the same struct declaration — plans 72-02/03 append to the var, not redeclare the type)
  - `var OrderingFixtures = []Fixture{ ... }` with entries sorted by (Entity, ID) for deterministic output

  - Use a regex-based line scanner over the upstream Python file; extract Django fixture blocks for network/organization/ix/ixlan/fac relevant to the ORDER category (rows with varying `updated` and `created` timestamps that drive the `(-updated, -created)` assertion).
  - If the upstream structure is ambiguous at boundary, write a permissive scanner: grep for `self.{entity} = {entity}.objects.create(` lines + the 5-10 lines following, extract the pk and key field values. Every extracted entry MUST carry an Upstream citation of form "pdb_api_test.py:<line>".
  - For this plan, hand-extract ≤15 ordering-relevant rows. Do not attempt full coverage; plans 72-02/03 expand.
