<empty_in_sentinel>
Two viable designs for the IN-02 empty `__in` short-circuit:

**A) Sentinel predicate** — `buildIn` returns `sql.FieldEQ("1", 0)` or equivalent "always false" predicate when `parts := strings.Split(value, ",")` yields `[]` or `[""]`. Pro: zero plumbing changes. Con: still issues a query (`WHERE 1=0`), which SQLite trivially elides but wastes a round-trip.

**B) QueryOptions.EmptyResult flag** — extend `QueryOptions` with `EmptyResult bool`; when `buildIn` detects empty, it sets `opts.EmptyResult = true` (via a side effect) and returns a harmless no-op predicate; the 13 list closures check `opts.EmptyResult` first and return `([]any{}, 0, nil)` WITHOUT any ent query. Pro: strictly correct no-SQL path. Con: threading via side-effect is ugly; better to add an `EmptyResult` return to `ParseFilters` itself.

**Decision: go with B, but thread it cleanly.** Change `ParseFilters` signature to:

```go
func ParseFilters(params url.Values, tc TypeConfig) (preds []func(*sql.Selector), emptyResult bool, err error)
```

The 13 list closures (registry_funcs.go) receive `emptyResult` via a new `QueryOptions.EmptyResult bool` field populated in `handler.go:serveList`. Each list closure's very first line becomes:

```go
if opts.EmptyResult {
    return nil, 0, nil
}
```

Handler serializes `{"data":[], "meta":{"count":0}}` just as it does for a successful empty list result (existing path — no handler changes beyond threading `EmptyResult` into `QueryOptions`).

Alternative: a middle-ground sentinel predicate that triggers WHERE 1=0 is acceptable if design B turns out to be a larger edit than anticipated. Document the choice taken in the SUMMARY.
</empty_in_sentinel>
