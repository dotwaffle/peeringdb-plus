**Important normalisation rule (applies to all 13):**

- Upstream uses Django's reverse-accessor suffix `_set` (e.g. `netfac_set__fac__name`). Our ent schemas declare forward edges without the `_set` suffix (see `ent/schema/network.go` Edges: `network_facilities`). Translate `<reverse>_set__<field>` upstream tokens to the matching local edge name by consulting the target schema's `Edges()` method. Document the translation in the source-line comment so an auditor can diff forward.
- Upstream occasionally exposes keys that would be >2 hops (e.g. `netfac_set__fac__country` is 2 hops in upstream — `netfac → fac → country` — but 3 underscores). Count the `__` separators carefully; our D-04 cap counts RELATION segments, not underscores. A field like `network_facilities__facility__country` has 2 relation segments (`network_facilities`, `facility`) plus a final field (`country`) and IS allowed.
- When in doubt, prefer dropping a borderline 2-hop over over-including: Plan 70-05's parser silently ignores unknown keys per D-05, so a missing allowlist entry produces a 200 with unfiltered results — an acceptable failure mode. Over-including an invalid path would cause a SQL join that doesn't resolve.
