- `import "peeringdb/v1/v1.proto";` (imports all 13 message types)

**For each of the 13 types, define:**

1. A `GetXxxRequest` message with `int64 id = 1;`
2. A `GetXxxResponse` message with `Xxx xxx = 1;` (using the message name from v1.proto)
3. A `ListXxxRequest` message with `int32 page_size = 1;` and `string page_token = 2;`
4. A `ListXxxResponse` message with `repeated Xxx xxx = 1;` and `string next_page_token = 2;`
5. A `service XxxService` with:
   - `rpc GetXxx(GetXxxRequest) returns (GetXxxResponse);`
   - `rpc ListXxx(ListXxxRequest) returns (ListXxxResponse);`

The 13 types and their exact message names from v1.proto are:
- Campus -> CampusService, GetCampus, ListCampuses
- Carrier -> CarrierService, GetCarrier, ListCarriers
- CarrierFacility -> CarrierFacilityService, GetCarrierFacility, ListCarrierFacilities
- Facility -> FacilityService, GetFacility, ListFacilities
- InternetExchange -> InternetExchangeService, GetInternetExchange, ListInternetExchanges
- IxFacility -> IxFacilityService, GetIxFacility, ListIxFacilities
- IxLan -> IxLanService, GetIxLan, ListIxLans
- IxPrefix -> IxPrefixService, GetIxPrefix, ListIxPrefixes
- Network -> NetworkService, GetNetwork, ListNetworks
- NetworkFacility -> NetworkFacilityService, GetNetworkFacility, ListNetworkFacilities
- NetworkIxLan -> NetworkIxLanService, GetNetworkIxLan, ListNetworkIxLans
- Organization -> OrganizationService, GetOrganization, ListOrganizations
- Poc -> PocService, GetPoc, ListPocs

Use `int32` for page_size (standard protobuf convention for counts/limits). Use `string` for page_token and next_page_token (opaque cursor). Use `int64` for id fields (matches v1.proto id field type).

Follow buf STANDARD lint rules. Service names must end with "Service" (buf.yaml service_suffix). RPC names must be PascalCase. Field names must be lower_snake_case. The response field name for the entity should be lower_snake_case of the message name (e.g., `Campus campus`, `CarrierFacility carrier_facility`).

Do NOT add any filter fields to ListXxxRequest -- that is Phase 24 (PROTO-04 only requires handler interfaces, not filtering). Keep request/response messages minimal and extensible.
