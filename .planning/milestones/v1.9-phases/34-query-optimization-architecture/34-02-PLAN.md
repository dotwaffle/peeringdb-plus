    **Create `internal/httperr/problem.go`:**

    Package `httperr` provides RFC 9457 Problem Details for HTTP API error responses.

    1. Define `ProblemDetail` struct with JSON tags:
       ```go
       type ProblemDetail struct {
           Type     string `json:"type"`
           Title    string `json:"title"`
           Status   int    `json:"status"`
           Detail   string `json:"detail,omitempty"`
           Instance string `json:"instance,omitempty"`
       }
       ```

    2. Define `WriteProblemInput` struct (per CS-5, >2 args):
       ```go
       type WriteProblemInput struct {
           Status   int
           Title    string
           Detail   string
           Instance string
       }
       ```

    3. Implement `WriteProblem(w http.ResponseWriter, input WriteProblemInput)`:
       - Set `Content-Type: application/problem+json`
       - Write HTTP status code
       - Encode `ProblemDetail` with `Type: "about:blank"`, `Title: input.Title` (or `http.StatusText(input.Status)` if empty), `Status: input.Status`, `Detail: input.Detail`, `Instance: input.Instance`
       - Use `json.NewEncoder(w).Encode(p)` for output
       - Document exported items per API-1

    4. Implement `NewProblemDetail(input WriteProblemInput) ProblemDetail` helper that constructs the struct without writing (for embedding in JSON responses like terminal mode):
       - Returns populated `ProblemDetail` struct
       - If `input.Title` is empty, uses `http.StatusText(input.Status)`

    **Create `internal/httperr/problem_test.go`:**

    Table-driven tests (per T-1):

    1. `TestWriteProblem` -- verify:
       - Status code is set on response
       - Content-Type is `application/problem+json`
       - Body contains `"type":"about:blank"`
       - Body contains correct `"status"` field
       - Body contains `"title"` field
       - Body contains `"detail"` field when non-empty
       - Body omits `"detail"` when empty (omitempty)
       - Body omits `"instance"` when empty

    2. `TestWriteProblem_DefaultTitle` -- verify that empty Title falls back to `http.StatusText()`.

    3. `TestNewProblemDetail` -- verify struct construction without writing.

    Use `httptest.NewRecorder()` for all tests. Mark tests with `t.Parallel()` per T-3.

    **Part A: Replace pdbcompat WriteError with httperr.WriteProblem (per ARCH-01 + CONTEXT.md decision)**

    In `internal/pdbcompat/response.go`:
    1. Replace the `WriteError` function body to call `httperr.WriteProblem`. Import `"github.com/dotwaffle/peeringdb-plus/internal/httperr"`.
    2. Keep the same function signature `WriteError(w http.ResponseWriter, status int, message string)` for backward compatibility within the package, but internally delegate to `httperr.WriteProblem(w, httperr.WriteProblemInput{Status: status, Detail: message})`.
    3. Remove the `errorMeta` struct and the old envelope-based error format. Successful responses (`WriteResponse`) remain unchanged.
    4. Keep the `X-Powered-By` header on error responses -- set it before calling `httperr.WriteProblem`.

    In `internal/pdbcompat/handler.go`:
    - No changes needed -- it already calls `WriteError()` which now delegates to httperr. However, add `Instance: r.URL.Path` to error calls where the request is available. Update the 6 `WriteError` call sites in handler.go to pass the request path. Change the internal `WriteError` signature to accept an optional instance string, or change call sites to use `httperr.WriteProblem` directly with the Instance field set to `r.URL.Path`. The cleaner approach: replace `WriteError` calls directly with `httperr.WriteProblem` calls, and remove `WriteError` from response.go. Set `Instance` to `r.URL.Path` at each call site.

    In `internal/pdbcompat/handler_test.go`:
    - Update error response assertions to expect RFC 9457 format: check for `"type":"about:blank"`, `"title"`, `"status"`, `"detail"` fields instead of `{"meta":{"error":"..."},"data":[]}` envelope.
    - This is an intentional API break per CONTEXT.md: "Apply to: PeeringDB compat".

    **Part B: Update web JSON error mode**

    In `internal/web/render.go`:
    1. Import `"github.com/dotwaffle/peeringdb-plus/internal/httperr"`.
    2. In the `ModeJSON` case (around lines 98-104), replace the ad-hoc error maps:
       - `map[string]any{"error": "not found", "status": 404}` becomes `httperr.NewProblemDetail(httperr.WriteProblemInput{Status: 404, Detail: "The page you're looking for doesn't exist."})`.
       - `map[string]any{"error": "internal server error", "status": 500}` becomes `httperr.NewProblemDetail(httperr.WriteProblemInput{Status: 500, Detail: "An unexpected error occurred."})`.
    3. Terminal text mode (ModeRich, ModePlain) error rendering via `renderer.RenderError()` remains unchanged -- it already has status/title/message structure.

    **Part C: REST error middleware**

    In `cmd/peeringdb-plus/main.go`:
    1. Import `"github.com/dotwaffle/peeringdb-plus/internal/httperr"`.
    2. Create a small middleware function `restErrorMiddleware` that wraps the entrest handler. It uses a `responseCapture` wrapper that captures the status code and body. If status >= 400, it replaces the response body with RFC 9457 format by calling `httperr.WriteProblem`. If status < 400, it passes through unchanged.
    3. Alternatively, since entrest may already write headers, a simpler approach: wrap the entrest handler with middleware that checks the response status code. Use an `http.ResponseWriter` wrapper that intercepts `WriteHeader` to capture the status, and `Write` to capture the body. When `WriteHeader(status >= 400)`, set Content-Type to application/problem+json and rewrite the error body.
    4. Apply this middleware: `mux.Handle("/rest/v1/", restCORS(restErrorMiddleware(restSrv.Handler())))`.
    5. The middleware MUST implement `http.Flusher` and `Unwrap() http.ResponseWriter` per CLAUDE.md Middleware conventions.

    **Part D: ConnectRPC -- no changes**
    ConnectRPC already uses `connect.NewError(code, err)` which produces structured errors. Per CONTEXT.md, we do NOT replace ConnectRPC's wire format with RFC 9457. No changes to `internal/grpcserver/`.
